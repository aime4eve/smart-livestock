## 一、评审结论

**目标**：系统同时支持 B2C（牧场主直订阅）和 B2B2C（平台→B端客户→最终用户）两种运营模式。

**结论**：当前技术方案（订阅服务计划 + 租户管理模块）在**功能权限控制层**设计完整且可直接复用，但在**商业模型层**（计费主体、用户体系、管理入口、部署隔离）存在系统性缺失。需要以现有技术骨架为基础，引入 `billingModel` 维度进行统一改造，分三个阶段落地。

## 二、商业模式定义

本产品面向 2B 场景，采用两种并行路径：

### 路径一：云托管分润模式

> 适用于接受 SaaS、愿意让利分成的场景

- 智能体产品统一部署在我方管理的亚马逊云/阿里云上
- B端客户直接向其最终用户（B或C端）收费
- 我方基于系统记录的用量或交易金额，与B端客户按分层合同定期分账
- 我方提供对账看板与自动结算接口
- 合同中约定数据审计权、最低保证收入及逾期处罚条款
- **增长逻辑**：持续收入来自终端使用量/流水的增长，驱动我们主动帮助 B端客户做好运营

### 路径二：独立部署订阅模式

> 适用于数据敏感、离线或买断倾向强烈的场景

- 按设备数量销售 License，捆绑年度软件订阅与支持服务（包含更新、远程运维及部分高级功能）
- 客户独立部署在自有环境内
- 通过 License 激活机制、定期远程健康检查、仅对订阅期客户提供新功能增购
- **增长逻辑**：收入来自 B端客户设备网络扩张 + 年度复购 + 新功能模块升级

### 落地保障措施

- 海外场景优先评估数据合规与跨境资金路径，合同中设计本地收款→服务费跨境支付结构
- 分润模式强制使用我方计费中心（网关）或埋点 SDK，确保收入数据一手可控
- 独立部署模式采用软 License 激活 + 在线心跳监控，防止超额使用

### 同时需要 B2C 直订阅

除上述两条 B2B2C 路径外，产品还需支持散户牧场主直接向平台按月付费订阅（B2C 模式），形成三种并行的商业通道：

| 通道 | 计费方 | 付费方 | 计费方式 |
|------|--------|--------|---------|
| B2C 直订阅 | 平台 | 牧场主 | 按月/按牲畜数 |
| B2B 云托管分润 | B端客户 | B端客户的最终用户 | 按用量/交易流水分账 |
| B2B 独立部署 | 平台 | B端客户 | 按设备 License 年付 |

## 三、现有技术方案评估

### 3.1 订阅服务计划：技术实现质量高，但规格仅覆盖 B2C

**计划 vs 规格覆盖度**：计划可以完整实现规格定义的全部需求。18 个 Task 拆分清晰，TDD 流程贯穿始终。

| 维度 | 覆盖度 | 说明 |
|------|--------|------|
| Response Shaping 架构 | **100%** | res.ok() 包装、filter→limit→lock 管道、ops 绕过 |
| 20 个 Feature Flag | **100%** | 全部定义，策略与规格一致 |
| 7 个订阅 API 端点 | **100%** | current/features/plans/checkout/cancel/renew/usage |
| 现有端点 shaping 接入 | **100%** | map/fences/alerts/dashboard/twin 全部注册 |
| 前端模型 + LockedOverlay | **100%** | SubscriptionTier 枚举、价格计算、锁定组件 |
| 完整流程 | **100%** | 试用→升级→续费→取消→降级→到期提醒 |
| Mock 模式兼容 | **100%** | applyMockShaping() 与后端逻辑等价 |

**核心判断**：规格本身是为 B2C 场景设计的（牧场主直接选套餐付费），计划忠实实现了规格。问题不在于计划质量，而在于规格的覆盖范围。

**4 处技术细节需注意**：

1. `daysUntilExpiry` 返回 999 而非 null（不影响业务逻辑）
2. `_lastDismissedDate` 用内存变量（MVP 简化，重启后重置）
3. `filterField` 用多字段 fallback（功能等价但不够精确）
4. `url_launcher` 依赖未确认（Task 10 执行前需验证）

### 3.2 租户管理模块：运维后台完整，但缺少前台自助能力

已实现能力：

| 能力 | 状态 |
|------|------|
| 租户 CRUD + 10 个 API 端点 | 已实现 |
| ops 管理后台（列表/创建/编辑/详情/趋势） | 已实现 |
| License 配额管理（licenseUsed / licenseTotal） | 已实现 |
| 3 角色权限体系（owner/worker/ops） | 已实现 |
| 分页/搜索/排序 | 已实现 |

### 3.3 关键洞察：Shaping 管道是业务模型无关的

Feature Flag + Response Shaping 架构只关心"这个租户能访问什么功能"，不关心"这个租户通过什么商业模式获得权限"。这意味着：

- 一个 B2B 客户通过合同获得 premium 权限
- 一个散户牧场主通过自助付费获得 premium 权限
- 在 Shaping 中间件层面，二者看到的 `tier=premium` 完全一样

**这 60% 的代码（Shaping 管道 + Feature Flags + LockedOverlay）三种商业模型共享，不需要改动。**

## 四、差距分析：从 B2C 到 B2C + B2B2C

以下按架构层从上到下分析缺失项。

### 4.1 用户体系层：扁平三角色 → 分层四角色

当前模型：

```
ops     → 平台运维（tenantId = null）
owner   → 租户管理员（tenantId = tenant_001）
worker  → 租户操作员（tenantId = tenant_001）
```

问题：`owner` 同时承担了"B端客户管理员"和"C端牧场主"两种角色，无法区分。

目标模型：

```
platform_admin  → 平台运维（当前 ops，不变）
b2b_admin       → B端客户管理员（新增，管理 B端下属的子用户和合同）
end_user        → 最终用户（当前 owner/worker 统一，按 billingModel 区分行为）
```

| 场景 | platform_admin | b2b_admin | end_user (owner) | end_user (worker) |
|------|:-:|:-:|:-:|:-:|
| B2C 直订阅 | 管理租户 | 不存在 | 自助注册/付费/管理牧场 | 继承 owner 权限 |
| B2B 分润 | 管理租户 | 管理合同/子用户/分润 | 由 b2b_admin 分配权限 | 同上 |
| B2B License | 管理租户 | 管理 License/心跳/增购 | 由 b2b_admin 分配权限 | 同上 |

### 4.2 租户数据模型层：缺少 billingModel 维度

当前 `tenantStore` 的 Tenant 模型：

```javascript
{ id, name, status, licenseUsed, licenseTotal, contactName, contactPhone, contactEmail, region, remarks }
```

需要新增：

| 字段 | 说明 | 适用路径 |
|------|------|---------|
| `billingModel` | `direct` / `revenue_share` / `licensed` | 全部 |
| `entitlementTier` | 当前授权层级（basic/standard/premium/enterprise） | 全部 |
| `contractId` | 关联合同 | 路径一、路径二 |
| `deploymentType` | `cloud` / `on_premise` | 路径二 |
| `licenseKey` | License 激活密钥 | 路径二 |
| `heartbeatAt` | 最后心跳时间 | 路径二 |

### 4.3 权限来源层：三种模型的数据源不同

| 维度 | B2C 直订阅 | B2B 分润 | B2B License |
|------|-----------|---------|-------------|
| **权限来源** | 用户自主选择套餐 | B端合同约定的模块+额度 | License 激活时写入的功能包 |
| **计费触发** | 用户点"升级"→Mock支付 | B端终端用户每次使用/交易 | B端采购时一次性/年度续费 |
| **前端升级入口** | 有（当前计划已覆盖） | 无（终端用户不选择套餐） | 无（由 B端管理员统一管理） |
| **价格展示** | 面向牧场主 ¥299/¥699 | 不面向终端展示 | 不面向终端展示 |
| **到期降级** | 用户到期自动降级 | 合同到期→B端整体降级 | License 到期→B端整体降级 |

**Shaping 中间件不需要改**——它只读取 `entitlementTier`，不关心 `billingModel`。

### 4.4 管理入口层：只有平台后台，缺少 B端和 C端入口

| 缺失入口 | 说明 |
|----------|------|
| **C端注册页** | B2C 牧场主自助注册→选套餐→创建租户+试用（当前仅 ops 可创建租户） |
| **C端"我的租户"页** | 牧场主查看自己租户的配额、状态、到期时间 |
| **B端管理员控制台** | B端客户登录后管理子用户、查看用量、配置功能 |
| **B端用户管理** | B端管理员创建/编辑/禁用自己的操作员和最终用户 |
| **B端用量看板** | B端客户查看设备用量、API 调用量、费用预估 |
| **B端合同信息页** | 查看当前合同、分润比例（路径一）或 License 状态（路径二）|

### 4.5 计量计费层：完全缺失

| 缺失能力 | 适用路径 | 说明 |
|----------|---------|------|
| API 调用计数 | 路径一 | 按租户统计 API 调用 |
| 交易流水记录 | 路径一 | 记录 B端客户向最终用户收费的流水 |
| 用量聚合 | 路径一 | 按日/周/月聚合用量数据 |
| 分润计算引擎 | 路径一 | 基于合同比例自动计算分账 |
| 对账看板 | 路径一 | B端客户自助查看分润明细 |
| 自动结算接口 | 路径一 | 定期分账执行 |
| License 生成/激活 | 路径二 | 生成 Key → B端激活 → 绑定设备数 |
| 心跳监控 | 路径二 | 定期健康检查 → 超额检测 → 告警 |
| License 到期处理 | 路径二 | 年度到期 → 续费提醒 → 过期降级 |

### 4.6 部署隔离层：单实例无法满足路径二

| 维度 | 当前 | 路径二需要 |
|------|------|-----------|
| 数据隔离 | 共享内存 Map + tenantId 过滤 | 独立数据库或 schema 级隔离 |
| 部署隔离 | 单实例 | 独立部署实例 |
| API 隔离 | 共享端点 | 专属 API 端点可选 |

## 六、分阶段落地路径

### Phase 1：统一基础设施（同时服务 B2C 和 B2B2C）

> 目标：建立 `billingModel` 维度，使现有技术骨架能够区分三种商业模型，完成 B2C 完整链路，为 B2B2C 扩展预留接口。

```
├── 权限控制层（不改）
│   └── Shaping 中间件、Feature Flags、LockedOverlay 保持现状
│
├── 数据模型扩展（小改）
│   ├── tenantStore：Tenant 新增 billingModel（默认 direct）、entitlementTier
│   ├── subscriptions.js：种子数据携带 billingModel: 'direct'
│   └── subscription_tier.dart：新增 BillingModel 枚举
│
├── API 层扩展（小改）
│   ├── /subscription/current 和 /plans 响应中携带 billingModel
│   └── /tenants 创建响应中携带 billingModel
│
├── 前端条件渲染（小改）
│   ├── SubscriptionStatusCard：billingModel=direct 显示升级按钮，其他显示"由管理员管理"
│   ├── SubscriptionPlanPage：仅 direct 可访问
│   └── LockedOverlay：不变（已经是数据驱动）
│
├── C端自助流程（新增）
│   ├── 注册页（注册→选套餐→创建租户+试用）
│   └── "我的租户"信息页（查看配额、状态、到期时间）
│
├── B2B2C 预留（新增空接口）
│   ├── ContractStore：定义接口但仅实现 direct 分支
│   └── LicenseStore：定义接口但仅实现 direct 分支
│
└── 测试
    ├── 现有订阅服务 18 Task 测试全部通过
    └── 新增 billingModel 相关断言
```

**工作量**：当前订阅服务计划的 +15%（含注册流程和 billingModel 扩展）

**交付物**：
- B2C 完整链路可用（注册→试用→选套餐→支付→使用→续费→取消）
- B2B2C 数据模型就绪（tenant 带 billingModel，接口预留）
- ops 管理后台可创建 direct / revenue_share / licensed 三种租户

### Phase 2：B2B2C 核心能力

## 八、最终建议

1. **立即执行当前订阅服务计划**，同时在实施过程中嵌入 `billingModel` 维度（+15% 工时）。这使得技术骨架从"仅服务 B2C"升级为"可区分三种模型"。

2. **Phase 1 交付后即可验证 B2C 完整链路**（注册→试用→付费→使用→续费→取消），同时 B2B2C 的数据模型和接口预留已就位。

3. **Phase 2 优先实现路径一（云托管分润）** 的计费中心——这是持续收入的核心引擎，包括用量计量、分润计算、对账结算。

4. **路径二（License）可与路径一并行开发**，其核心模块（License 生成/激活/心跳）与分润引擎无耦合关系。

5. **海外合规从 Phase 1 开始在合同模板中预留条款**，技术实现放在 Phase 3。

6. **不建议推迟 `billingModel` 到 Phase 2**——如果在 Phase 1 不埋入这个维度，Phase 2 扩展时需要全面重构数据模型和 API 响应，代价远高于现在多花 15% 工时。
