## 二、数据基础设施需求

### 2.1 传感器数据采集规格

ML 模型依赖以下三类传感器持续采样的数据。当前 Demo 阶段由时序生成器模拟，MVP 阶段需由真实 IoT 设备提供。

#### 2.1.1 GPS 追踪器

| 参数 | 规格 | 采样频率 | 对应模型字段 |
|------|------|----------|-------------|
| 经度 (lng) | WGS-84，精度 2-5m | 5-15 min/次 | `GeoPoint.lng` |
| 纬度 (lat) | WGS-84，精度 2-5m | 5-15 min/次 | `GeoPoint.lat` |
| 时间戳 | ISO 8601 UTC | 同上报周期 | `GeoPoint.timestamp` |

**ML 诉求**：需提供连续轨迹序列，支持 24h / 7d / 30d 多时间窗口查询。

#### 2.1.2 瘤胃胶囊

| 参数 | 规格 | 采样频率 | 对应模型字段 |
|------|------|----------|-------------|
| 瘤胃温度 | 精度 ±0.1°C，范围 36-42°C | 30 min/次 | `TemperatureRecord.temperature` |
| 蠕动频率 | 次/分钟，范围 0-2.2 | 30 min/次 | `MotilityRecord.frequency` |
| 蠕动强度 | 无量纲，范围 0-1.0 | 30 min/次 | `MotilityRecord.intensity` |

**ML 诉求**：30 分钟粒度已满足体温趋势与蠕动节律分析；若硬件支持更高频率（如 5 min），可用于更精细的行为模式挖掘。

#### 2.1.3 加速度计

| 参数 | 规格 | 采样频率 | 对应模型字段 |
|------|------|----------|-------------|
| 步数 | 累计步数 | 上报时计算 | `LivestockDetail.activityLevel`（派生） |
| 活动等级 | 分类标签 | 上报时计算 | `LivestockDetail.activityLevel` |

**ML 诉求**：边缘端需完成原始三轴加速度 → 步数 → 活动等级的预处理。云端接收聚合后的步数与活动等级，用于发情检测与健康评分。

### 2.2 数据质量要求

| 维度 | 要求 | 影响场景 |
|------|------|----------|
| **时间连续性** | 单个传感器 24h 内数据缺失率 ≤ 10% | 所有时序分析 |
| **温度精度** | 瘤胃温度测量误差 ≤ ±0.1°C | 发热预警、健康评分 |
| **GPS 精度** | 定位误差 ≤ 5m（开窗区域） | 围栏告警、接触链路 |
| **时钟同步** | 所有传感器时钟偏差 ≤ 30s | 多传感器融合时间对齐 |
| **异常值过滤** | 传感器故障值（如温度 = 0 或 > 50°C）需在边缘端过滤 | 数据清洗 |

### 2.3 特征工程需求

| 特征类别 | 具体特征 | 计算窗口 | 消费场景 |
|----------|----------|----------|----------|
| **体温特征** | 个体基线温度（7d 均值）、昼夜温差、24h/72h 趋势斜率、偏离度 Δ | 7d / 72h / 24h | ML-1, ML-5 |
| **蠕动特征** | 个体基线蠕动频率（7d 均值）、蠕动频率下降百分比、零蠕动事件计数 | 7d / 24h | ML-2, ML-5 |
| **活动特征** | 日均步数、步数变化百分比、活动等级分布（静止/行走/奔跑占比） | 24h / 7d | ML-3, ML-5, ML-6 |
| **位置特征** | 日均移动距离、活动半径、围栏边界接近频率、放牧区域偏好 | 24h / 7d | ML-3, ML-7 |
| **群体特征** | 群体平均体温、群体平均活动量、异常个体占比、群体体温趋势 | 实时 / 24h | ML-4 |
| **交叉特征** | 体温↑ + 活动↓ 组合指标、体温↑ + 蠕动↓ 组合指标、步数↑ + 距离↑ + 体温↑ 组合指标 | 实时 | ML-1, ML-3, ML-5 |

## 三、ML 需求详细定义

### ML-1：体温异常预警（发热预警）

#### 3.1.1 业务背景

瘤胃温度是牲畜健康的**核心指标**。当前 Demo 已实现个体体温基线建模与 72 小时趋势展示（`twin_models.dart: TemperatureBaseline`）。MVP 阶段需将基于固定阈值的规则升级为基于 ML 的自适应异常检测。

#### 3.1.2 输入数据

| 数据项 | 来源 | 粒度 | 时长 |
|--------|------|------|------|
| 历史体温序列 | 瘤胃胶囊 | 30 min | ≥ 7 天 |
| 个体元数据 | 牲畜档案 | — | 品种、年龄（月）、体重 |
| 环境温度（可选） | 气象站 | 1h | 同上 |

**输入特征向量**（单个预测样本）：

```
X = [baseline_temp, current_temp, delta_24h, delta_72h,
     trend_slope_72h, max_temp_24h, circadian_deviation,
     breed_encoded, age_months, weight_kg]
```

#### 3.1.3 输出结果

| 输出项 | 类型 | 说明 | 对应模型字段 |
|--------|------|------|-------------|
| 状态分级 | `normal` / `warning` / `critical` | 三级分类 | `TemperatureBaseline.status` |
| AI 判断结论 | 文本 | 中文描述性结论 | `TemperatureBaseline.conclusion` |

**分级规则（业务逻辑）**：

| 状态 | 触发条件 | 典型结论模板 |
|------|----------|-------------|
| normal | Δ ≤ 0.5°C 且无持续上升趋势 | "体温稳定，未见异常波动" |
| warning | 0.5°C < Δ ≤ 1.0°C 或持续上升趋势 | "体温轻度升高，建议持续观察饮水与采食" |
| critical | Δ > 1.0°C 或温度升高 + 活动量下降 | "温度升高+活动量下降，高概率感染，建议隔离检查" |

> 注：以上为 Demo 阶段的硬编码规则。ML 模型训练后，分级应由模型输出概率 + 阈值决定，规则保留作为兜底。

#### 3.1.4 多信号融合判断逻辑

当前 Demo 中 `conclusion` 的生成逻辑暗示了多信号融合需求：

```
IF 温度升高 AND 活动量下降 → 高概率感染 → critical
IF 温度升高 AND 蠕动下降   → 消化系统疾病 → critical（关联 ML-2）
IF 温度轻度升高 AND 其他指标正常 → 持续观察 → warning
IF 温度稳定               → 正常 → normal
```

**ML 需求**：需构建多信号融合分类器，输入为 {体温特征, 活动特征, 蠕动特征}，输出为 {normal, warning_infection, warning_digestive, critical_infection, critical_digestive}。

#### 3.1.5 性能要求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 灵敏度（召回率） | ≥ 90% | 漏检一头发热牲畜的代价远大于误报 |
| 特异性 | ≥ 80% | 误报过多会降低用户信任 |
| 推理延迟 | ≤ 1s | 单头个体预测 |
| 基线学习周期 | ≤ 7 天 | 新个体入群后 7 天内建立可靠基线 |
| 日节律适应 | 需要 | 牛体温存在昼夜波动（白天高 0.2°C），基线需区分昼夜 |

#### 3.1.6 验收标准

1. 在标注数据集上，critical 级别的召回率 ≥ 90%，precision ≥ 75%
2. 连续 72 小时体温曲线中的异常事件检测延迟 ≤ 2 小时
3. 新个体入群 7 天后，基线稳定度（7d 前后均值偏差）≤ 0.2°C
4. 误报率（健康牲畜被标记为 warning/critical 的比例）≤ 15%/天

### ML-2：消化健康监测

#### 3.2.1 业务背景

消化系统疾病占牛群疾病的 30-40%。当前 Demo 已实现蠕动频率基线建模与 24 小时蠕动趋势展示（`twin_models.dart: DigestiveHealth`）。瘤胃胶囊提供的蠕动数据（频率 + 强度）是消化健康评估的核心输入。

#### 3.2.2 输入数据

| 数据项 | 来源 | 粒度 | 时长 |
|--------|------|------|------|
| 历史蠕动频率序列 | 瘤胃胶囊 | 30 min | ≥ 7 天 |
| 历史蠕动强度序列 | 瘤胃胶囊 | 30 min | ≥ 7 天 |
| 个体元数据 | 牲畜档案 | — | 品种、年龄、体重 |

**输入特征向量**：

```
X = [motility_baseline, current_freq, freq_drop_percent,
     current_intensity, zero_motility_events_24h,
     freq_variance_24h, breed_encoded, age_months]
```

#### 3.2.3 输出结果

| 输出项 | 类型 | 说明 | 对应模型字段 |
|--------|------|------|-------------|
| 状态分级 | `normal` / `warning` / `critical` | 三级分类 | `DigestiveHealth.status` |
| 处置建议 | 文本 | 中文建议 | `DigestiveHealth.advice` |

**分级规则**：

| 状态 | 触发条件 | 典型建议模板 |
|------|----------|-------------|
| normal | 蠕动频率在基线 ±20% 内 | "蠕动节律正常" |
| warning | 蠕动频率下降 40-60% | "蠕动频率下降，建议检查饲粮与饮水" |
| critical | 蠕动频率下降 > 60% 或完全停止 | "蠕动完全停止，疑似瘤胃臌气，需立即处理" |

#### 3.2.4 蠕动节律建模需求

当前 Demo 的 `MotilityGenerator` 揭示了正常牛的蠕动节律模式：

| 时段 | 正常蠕动频率（次/min） | 说明 |
|------|---------------------|------|
| 采食高峰 (6-10, 16-20) | 1.1 - 1.55 | 进食驱动蠕动增强 |
| 夜间休息 (23-5) | 0.72 - 1.04 | 反刍期蠕动减弱 |
| 其他时段 | 0.92 - 1.34 | 正常放牧活动 |

**ML 需求**：模型需学习个体的日节律模式（daytime pattern），异常检测应对比**同时段基线**而非全局基线。

#### 3.2.5 性能要求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| critical 级别召回率 | ≥ 95% | 蠕动停止是紧急状况，不能漏检 |
| warning 级别召回率 | ≥ 85% | |
| 推理延迟 | ≤ 1s | |
| 零蠕动事件检测延迟 | ≤ 1 小时 | 从蠕动停止到告警触发 |

#### 3.2.6 验收标准

1. 蠕动完全停止（连续 ≥ 2 小时频率 = 0）检出率 ≥ 98%
2. 蠕动频率下降 50% 以上的检出延迟 ≤ 2 小时
3. 正常采食高峰期蠕动增强不被误判为异常

### ML-3：发情识别与配种建议

#### 3.3.1 业务背景

发情漏检是繁殖效率损失的主要来源。当前 Demo 已实现多传感器融合的发情评分模型（`twin_models.dart: EstrusScore`），综合步数增加、体温变化、移动距离变化三维度给出 0-100 分。一头奶牛每天非产奶损失 30-50 元，延迟 21 天配种损失 630-1050 元。

#### 3.3.2 输入数据

| 数据项 | 来源 | 粒度 | 时长 |
|--------|------|------|------|
| 步数增加百分比 | 加速度计（边缘聚合） | 日 | 7 天 |
| 体温变化量（Δ） | 瘤胃胶囊 | 日均值 | 7 天 |
| 移动距离变化量（Δ） | GPS | 日累计 | 7 天 |
| 发情评分趋势 | ML 模型输出 | 日 | 7 天 |
| 个体繁殖历史 | 牲畜档案 | — | 上次配种日期、产犊次数 |

**输入特征向量**：

```
X = [step_increase_percent, temp_delta, distance_delta,
     score_7d_trend, score_3d_avg, score_peak_7d,
     days_since_last_calving, breed_encoded, age_months]
```

#### 3.3.3 输出结果

| 输出项 | 类型 | 范围 | 对应模型字段 |
|--------|------|------|-------------|
| 发情综合评分 | 整数 | 0-100 | `EstrusScore.score` |
| 步数增加百分比 | 整数 | ≥ 0% | `EstrusScore.stepIncreasePercent` |
| 体温变化量 | 浮点 | °C | `EstrusScore.tempDelta` |
| 距离变化量 | 浮点 | km | `EstrusScore.distanceDelta` |
| 配种建议 | 文本 | 中文 | `EstrusScore.advice` |
| 7 日评分趋势 | 时序 | `EstrusTrendPoint[]` | `EstrusScore.trend7d` |

#### 3.3.4 评分模型需求

当前 Demo 的发情评分呈现典型的 21 天发情周期模式（`EstrusScoreGenerator`）：

| 发情周期天数 | 典型评分范围 | 状态 |
|-------------|-------------|------|
| Day 1-16（间情期） | 10-30 | 非发情 |
| Day 17-18（发情前期） | 40-60 | 信号增强 |
| Day 19-20（发情期） | 75-100 | 发情高峰 |
| Day 21（过渡） | 10-30 | 周期结束 |

**ML 需求**：

1. **多特征融合评分**：将步数增加百分比、体温变化、距离变化三个维度融合为单一发情评分（0-100）
2. **周期模式识别**：学习个体发情周期（约 21 天），预测下次发情窗口
3. **配种时机推荐**：评分 ≥ 80 时推荐 6 小时内配种；评分 60-80 时推荐 12 小时内关注

#### 3.3.5 各维度典型发情信号

| 维度 | 非发情期 | 发情期变化 | Demo 中数据 |
|------|---------|-----------|-----------|
| 步数 | 正常（~2000 步/日） | 增加 180%+ | `stepIncreasePercent: 180-380` |
| 体温 | 基线 ± 0.1°C | 升高 0.2-0.5°C | `tempDelta: 0.2-0.4` |
| 移动距离 | 正常 | 增加 1.5-4.5 km | `distanceDelta: 1.5-4.5` |

#### 3.3.6 性能要求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 发情检出率（召回率） | ≥ 85% | 漏检一个发情周期 = 损失 21 天 |
| 误报率 | ≤ 10%/周期 | 过多误报消耗人工配种资源 |
| 评分准确率 | ±15 分以内 | 模型评分与兽医标注评分偏差 |
| 配种时机推荐准确率 | ±6 小时 | |
| 周期预测精度 | ±2 天 | 预测下次发情日期 |

#### 3.3.7 验收标准

1. 在标注的发情周期数据中，检出率 ≥ 85%（评分 ≥ 60 视为检出）
2. 非发情期日均误报率 ≤ 5%
3. 发情高峰期（评分 ≥ 80）检出延迟 ≤ 12 小时
4. 7 日评分趋势图与人工标注的发情曲线趋势一致（Pearson ≥ 0.8）

### ML-4：群体疫病防控

#### 3.4.1 业务背景

疫病爆发可能导致全群扑杀，损失数百万元。当前 Demo 已实现群体健康指标监控与 GPS 接触链路追踪（`twin_models.dart: HerdHealthMetrics`, `ContactTrace`）。

#### 3.4.2 输入数据

| 数据项 | 来源 | 粒度 | 时长 |
|--------|------|------|------|
| 群体平均体温 | 瘤胃胶囊聚合 | 1h | 30 天 |
| 群体平均活动量 | 加速度计聚合 | 1h | 30 天 |
| 异常个体占比 | ML-1 + ML-5 输出 | 日 | 30 天 |
| GPS 接触矩阵 | GPS 坐标对计算 | 15 min | 7 天 |
| 个体异常状态 | ML-1 输出 | 实时 | — |

#### 3.4.3 输出结果

| 输出项 | 类型 | 说明 | 对应模型字段 |
|--------|------|------|-------------|
| 群体健康指标 | 聚合数据 | 平均体温、平均活动量、异常率 | `HerdHealthMetrics` |
| 群体状态 | `normal` / `elevated` / `alert` | 系统性偏移预警 | `SceneSummaryEpidemic.status` |
| 接触链路图 | 有向图 | 潜在传播路径 | `ContactTrace[]` |
| 隔离优先级列表 | 排序列表 | 按接触频率与接近距离排序 | 隐含于 `ContactTrace.proximity` |

#### 3.4.4 接触链路追踪需求

当前 Demo 的 `ContactTrace` 模型定义了以下数据结构：

```
ContactTrace {
  fromId: String       // 传播源个体 ID
  toId: String         // 接触个体 ID
  lastContact: DateTime // 最近接触时间
  proximity: Double    // 接近距离（米）
}
```

**ML 需求**：

1. **接触图构建**：基于 GPS 坐标，计算所有个体对之间的距离，阈值内（如 ≤ 15m）视为接触
2. **传播概率估算**：结合接触持续时间、接近距离、源个体异常程度，估算传播概率
3. **隔离优先级排序**：根据 {接触频率, 接近距离, 传播概率} 综合排序

#### 3.4.5 群体异常检测需求

**检测场景**：

| 场景 | 定义 | 检测方法 |
|------|------|----------|
| 群体体温偏移 | 群体平均体温在 24h 内升高 > 0.3°C | 时序异常检测 |
| 异常个体激增 | 异常个体占比突然升高（如 2% → 8%） | 统计过程控制（SPC） |
| 空间聚集异常 | 异常个体在地理上呈聚集分布 | 空间聚类分析 |
| 传播链预警 | 接触链中出现级联传播模式 | 图异常检测 |

#### 3.4.6 性能要求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 群体异常检测延迟 | ≤ 4 小时 | 从群体指标开始偏移到告警触发 |
| 接触链计算延迟 | ≤ 5 min | 50 头牲畜的全体对距离计算 |
| 传播概率估算准确率 | ≥ 70% | 与兽医流行病学调查对比 |
| 隔离优先级排序准确率 | ≥ 80% | Top-5 隔离对象中至少 4 头为真实高风险个体 |

#### 3.4.7 验收标准

1. 群体平均体温偏移 0.5°C 时，4 小时内触发预警
2. 异常个体占比从基线升高 5 个百分点时，触发群体告警
3. 接触链路图在地图可视化中正确展示传播路径
4. 隔离优先级列表与兽医人工判断 Top-5 重合度 ≥ 80%

#### 3.5.4 当前 Demo 中的健康数据模式

1. 健康个体评分 ≥ 80 的比例 ≥ 90%
2. 异常个体评分 ≤ 50 的比例 ≥ 90%

### ML-6：行为异常检测

#### 3.6.1 业务背景

当前 Demo 的告警系统中已包含"行为异常"类型（`alert-010`, `alert-012`, `alert-014`，type = `behavior`），但尚未定义行为异常的具体检测逻辑。ML-6 需要从加速度计数据中识别异常行为模式。

#### 3.6.2 输入数据

| 数据项 | 来源 | 粒度 |
|--------|------|------|
| 加速度序列（三轴） | 加速度计 | 高频（边缘预处理） |
| 步数时间序列 | 加速度计（边缘聚合） | 5-15 min |
| 活动等级序列 | 加速度计（边缘分类） | 5-15 min |
| GPS 移动速度 | GPS | 5-15 min |

#### 3.6.3 输出结果

| 输出项 | 类型 | 说明 |
|--------|------|------|
| 行为异常评分 | 浮点 (0-1) | 异常程度 |
| 异常类型标签 | 枚举 | `lethargy`（嗜睡）/ `restless`（烦躁）/ `lameness`（跛行）/ `isolated`（离群） |
| 告警级别 | 枚举 | P2（当前 Demo 定义） |

#### 3.6.4 异常行为模式定义

| 异常类型 | 行为特征 | ML 检测方法 |
|----------|----------|-----------|
| 嗜睡 (lethargy) | 白天活动量骤降、步数 < 日均 30%、长时间静止 | 活动量时序异常检测 |
| 烦躁 (restless) | 频繁走停交替、无目的地移动、步数 > 日均 200% | 步态模式识别 |
| 跛行 (lameness) | 步态不对称、移动速度下降、步幅异常 | 步态分析（需高频加速度数据） |
| 离群 (isolated) | GPS 位置偏离群体中心 > 2km 持续 > 2h | 空间聚类离群检测 |

#### 3.6.5 性能要求

| 指标 | 目标值 |
|------|--------|
| 行为异常检出率 | ≥ 80% |
| 误报率 | ≤ 10%/天 |
| 检出延迟 | ≤ 2 小时 |

#### 3.6.6 验收标准

1. 嗜睡（活动量下降 > 70% 持续 > 4 小时）检出率 ≥ 85%
2. 离群（偏离群体 > 2km 持续 > 2h）检出率 ≥ 90%
3. 正常放牧行为的误报率 ≤ 5%/天
