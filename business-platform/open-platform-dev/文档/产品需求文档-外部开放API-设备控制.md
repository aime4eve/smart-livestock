## 1. 背景与目标

### 1.1 背景

波兰客户需要其应用作为**外部客户端**，通过标准 REST 接口与本公司设备管理侧对接：采集状态与日志、下发开关等指令。公司内部已有 AI 智能体相关能力，包含**网关、各类中台及 MVP 业务系统**，需在现有网关上做**定制化开发**，对外暴露统一、可文档化的 API。

### 1.2 目标

- 提供与客户规范一致的 **HTTPS + JSON** REST API。
- 实现清晰的**双层凭证模型**：集成方凭 **appId / appSecret** 管理 API Key；凭 **API Key** 调用设备与业务接口。
- **API Key** 满足长度、存储（哈希）、**Scope** 与 HTTP 方法校验等安全要求。
- **一期**交付客户规范中的 **认证与密钥管理、读数据、控制及辅助/Admin（第 3–6 节）** 全部能力；**仅沙箱环境**不在一期（见 3.2）。

### 1.3 成功标准（摘要）

- 客户可凭书面发放的 **appId、appSecret** 完成密钥的创建、列表查询、吊销与轮换。
- 客户可凭 **API Key** 完成文档所列读、写及 **admin** 管理类接口调用，且权限与 Scope 一致。
- 提供完整**对外接口说明文档**；**沙箱环境**不在本期范围（见 3.2），需在对外说明中明确。

## 2. 术语

| 术语 | 说明 |
|------|------|
| 集成方 / 客户 | 调用开放 API 的外部应用方 |
| appId / appSecret | 由本公司**人工发放**的一对集成凭证，用于调用**密钥管理类接口** |
| API Key（api_key） | 通过密钥管理接口创建、用于**设备与业务接口**的访问密钥；明文仅在创建或轮换成功时返回一次 |
| key_id | 服务端生成的 API Key 记录标识，用于列表展示、吊销、轮换 |
| Scope | 单把 API Key 的权限范围；`read_write` 表示**一把 Key 同时具备读与写类权限**（非两把 Key） |
| 空间（space） | 设备的物理或逻辑分组单元（如房间、楼层、区域），每个空间有唯一标识 `space_id`；空间数据由内部**空间中台**提供，网关做映射与透传 |

## 3. 范围

### 3.1 本期包含

- 通用约定：REST、JSON、标准 HTTP 状态码、错误体、分页与过滤（见第 5 节非功能需求）。
- **认证与授权**：双层凭证、Scope 与 HTTP 方法绑定、密钥哈希存储。
- **密钥管理**：`POST/GET/DELETE/PUT …/api-keys` 全链路（含列表分页、吊销、轮换）。
- **空间管理**：空间的查询、创建、修改、删除；空间数据来源于内部空间中台，网关做映射与透传。
- **设备数据读取**：列表、详情、活动、汇总等（路径与字段以客户规范为基线，可与内部 MVP 对齐实现）。
- **设备控制**：下发命令、设置、取消命令、查询命令状态等。
- **辅助与 Admin**（客户规范第 6 节）：注册设备、删除设备等，与 **第 8、9 节同属一期**。
- **对外文档**：接口说明（含鉴权方式、示例、错误码说明）。

### 3.2 本期不包含（需对外声明）

- **沙箱 / 独立测试环境**：客户原始需求中的 sandbox；**一期不建设**，需在《对外接口说明》中明确；若后续建设可单独立项。

### 3.3 假设与依赖

- 设备与指令的实际数据来源、下发通道由现有**中台 / MVP** 提供；网关负责鉴权、路由、与内部服务编排。
- 客户接受本期无独立沙箱，或使用**约定限流的联调环境**替代（若商务另行提供，不在本文档展开）。

## 4. 用户角色与典型场景

| 角色 | 能力 |
|------|------|
| 集成方开发者 | 使用 appId/appSecret 管理 API Key；使用 API Key 调试读/写接口 |
| 运维 / 安全 | 吊销泄露 Key、轮换 Key；审计 key_id 维度使用情况（若内部有日志则支持排查） |

**典型流程**

1. 本公司向客户发放 **appId、appSecret** 与《对外接口说明》。
2. 客户调用 **POST `/api-keys`** 创建 Key，获得 **api_key**（仅展示一次）与 **key_id** 等。
3. 客户使用 **GET `/api-keys`**（带 appId/appSecret）查看当前名下所有 Key 元数据（不含完整明文密钥）。
4. 客户在业务请求头中携带 **API Key**，调用 `/devices` 等接口。
5. 需要时调用 **DELETE** 吊销或 **PUT …/rotate** 轮换；轮换后旧明文失效，新明文仅在新响应中出现一次。

## 5. 非功能需求

### 5.1 协议与格式

- **HTTPS** 强制；全链路请求与响应体为 **JSON**（除文档另行约定外）。
- **HTTP 方法语义**：GET 查询、POST 创建/命令、PUT 更新、DELETE 删除；与业界惯例一致。
- **HTTP 状态码**：使用 200、201、400、401、403、404、500 等标准语义；业务错误时仍返回结构化 JSON（见下）。

### 5.2 错误响应

错误时返回 JSON，至少包含可读信息，建议形态：

```json
{
  "error": "简短错误码或说明",
  "details": "可选，供排障的补充说明"
}
```

### 5.3 列表与过滤

- 列表类接口支持 **`limit`、`offset`** 分页。
- 支持通过 **Query** 做过滤；**设备列表**中与空间相关的参数：客户规范参数名为 **`location`**，在本产品中**统一定义为「按空间筛选」**——传入值为 **`space_id`**（空间唯一标识，与内部空间中台 ID 映射一致；客户示例 `room1` 仅作样例）。空间为独立实体，支持查询、创建、修改、删除，详见 **第 8.2 节**及**第 11 节**。
- 各接口具体 Query 以 **第 7–9 节、第 8.2 节及**《对外接口说明》为准。

### 5.4 安全

- **API Key**：长度不少于 **32** 字符，建议字母数字组成；**数据库仅存哈希**（如 bcrypt / argon2），校验时使用安全比对。
- **appSecret**：仅存哈希；传输仅在 HTTPS 下进行。
- **密钥管理接口**与**业务接口**鉴权方式区分清晰，避免混用导致越权。
- 网关对密钥管理接口实施合理 **限流** 与 **审计日志**（内部要求，对外文档可简述）。

## 6. 功能需求：认证与授权模型

### 6.1 双层凭证

| 接口类型 | 鉴权方式 | 说明 |
|----------|----------|------|
| 密钥管理：`/api-keys` 及同前缀管理类接口 | **appId + appSecret** | 由本公司发放；用于创建/列出/吊销/轮换 API Key；**仅作用于该 app 名下的 Key** |
| 设备与业务：如 `/devices`、`/activity`、`/commands` 等 | **API Key** | 请求头：`X-API-Key: <api_key>` 或 `Authorization: Bearer <api_key>` |

**约束**

- **不得**将 appSecret 放在 URL 查询参数中。
- 建议使用 **HTTP Basic**（用户名 appId、密码 appSecret）或等价的 **Authorization: Basic** / 专用头组合；具体格式在《对外接口说明》中固定一种，避免多实现并存。

### 6.2 Scope 定义与 HTTP 方法绑定

单把 API Key 对应 **一个** scope 字段（创建时由客户端传入，服务端校验枚举）。与 HTTP 方法的对应关系如下（可在实现中微调 DELETE 的归属，但须在对外文档中写死）：

| Scope | 允许的操作（摘要） |
|-------|---------------------|
| `read` | 仅 **GET**（读设备、读活动、读汇总等） |
| `write` | **POST/PUT** 等写操作（下发命令、改设置等）；是否包含部分 GET 由产品决定，**建议 write 不包含仅读以外的 DELETE** |
| `read_write` | **同时具备** `read` 与 `write` 允许的方法（**一把 Key**，非两把 Key） |
| `admin` | 全量含 **DELETE** 设备、注册设备等管理类接口（见第 10 节，**一期交付**） |

**网关行为**：每个业务请求根据 Key 的 scope 与 HTTP 方法判定；不满足则返回 **403 Forbidden**。

### 6.3 与密钥展示相关的规则

- **创建（POST）与轮换（PUT rotate）** 的响应中可返回 **明文 api_key**（或 `new_api_key`），**仅该次响应**。
- **列表（GET）** 仅返回 **key_id、description、scope、created_at、last_used** 等元数据，**不返回**完整 api_key 明文。

## 7. 功能需求：密钥管理 API

以下路径以**客户规范为参考**；若网关统一加前缀（如 `/v1`），在对外文档中写清**完整路径**。

### 7.1 POST `/api-keys` — 生成新 Key

**鉴权**：appId + appSecret  

**请求体（示例字段）**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| description | string | 否 | 描述 |
| scope | string | 是 | 枚举：`read`、`write`、`read_write`、`admin`（以对外文档为准） |
| expires_in_days | number | 否 | 有效期（天）；若不传则按产品策略默认或永不过期（需二选一并在文档说明） |

**响应（201）**：包含 `api_key`（明文一次）、`key_id`、`created_at`、`scope` 等。

### 7.2 GET `/api-keys` — 列出当前 app 下所有 Key

**鉴权**：appId + appSecret  

**查询参数**：`limit`、`offset`（分页）

**响应（200）**：**数组**；每项为一把 Key 的元数据（**多个 key_id 表示多把 Key**，非一个 key_id 对应多串明文）。字段与客户规范对齐，如：`key_id`、`description`、`scope`、`created_at`、`last_used`。

### 7.3 DELETE `/api-keys/{key_id}` — 吊销

**鉴权**：appId + appSecret  

**约束**：`key_id` 必须属于当前 app；否则 404 或 403（对外文档统一一种）。

**响应（200）**：如 `{ "status": "revoked" }`。

### 7.4 PUT `/api-keys/{key_id}/rotate` — 轮换

**鉴权**：appId + appSecret  

**行为**：生成新明文 Key，**作废**旧 Key；新明文仅在本次响应返回。

**响应（200）**：如 `{ "new_api_key": "..." }`，并返回或隐含新 `key_id` 规则（若与客户规范一致则仅更新同一 `key_id` 的密钥材料，以技术实现为准并在文档说明）。

## 8. 功能需求：设备读（Read）

覆盖客户规范第 4 节核心能力，包括但不限于：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/devices` | 设备列表；支持过滤、分页（见 8.2） |
| GET | `/devices/{device_id}` | 设备详情与指标；**无**客户规范所列 Query，仅路径参数 |
| GET | `/devices/{device_id}/activity` | 活动历史；日期范围 Query（见 8.2） |
| GET | `/activity/summary` | 活动汇总；`period` Query（见 8.2） |

### 8.2 客户规范中的查询、分页及「空间」约定

**`location` = 按空间筛选（`space_id`）**
与客户 HTML 保持一致，查询参数名使用 **`location`**；语义为**仅返回该空间下的设备**。取值类型为字符串，值为 **`space_id`**（空间唯一标识，由空间管理接口创建或维护；与内部空间中台 ID 映射一致；客户示例 `room1` 仅作样例）。未传时不按空间过滤（仍受租户、数据权限约束）。空间管理接口详见**第 11 节**。

| 接口 | Query / 分页（客户 HTML） | 说明 |
|------|---------------------------|------|
| `GET /devices` | `status`、`location`、**`limit`**；并支持通用 **`offset`**（与客户第 1 节「列表分页」一致） | `status` 过滤设备状态；**`location` 即空间**；分页与 §5.3 一致 |
| `GET /devices/{device_id}` | （客户未列 Query） | 仅 `device_id` 路径参数 |
| `GET /devices/{device_id}/activity` | `from`、`to` | 日期格式 **`YYYY-MM-DD`**，表示活动历史时间范围 |
| `GET /activity/summary` | `period` | 取值 **`day`** / **`week`** / **`month`**（客户 HTML 明示） |

**鉴权**：API Key；Scope 需包含读权限（`read` 或 `read_write` 或 `admin`，以 6.2 为准）。

响应 JSON 结构与客户示例对齐；字段与内部 MVP 数据模型不一致时，在网关或 BFF 层做映射并在文档中列出字段说明。

## 9. 功能需求：设备写与控制（Write）

覆盖客户规范第 5 节核心能力，包括但不限于：

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/devices/{device_id}/commands` | 下发命令 |
| PUT | `/devices/{device_id}/settings` | 更新设置 |
| DELETE | `/devices/{device_id}/commands/{command_id}` | 取消命令 |
| GET | `/commands/{command_id}/status` | 查询异步命令状态（轮询） |

**鉴权**：API Key；Scope 需包含写权限（`write` 或 `read_write` 或 `admin`）。

**与客户规范第 5 节的补充说明**：该节接口以 **JSON 请求体** 为主（如 `action`、`params`、`settings`），客户 HTML **未再约定**额外 Query 参数；轮询类 `GET` 仅路径中的 `command_id`。

## 10. 功能需求：辅助与 Admin

对应客户规范第 6 节，**一期交付**：

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/devices` | 注册新设备（请求体见下） |
| DELETE | `/devices/{device_id}` | 删除设备 |

**鉴权**：API Key，Scope 为 **`admin`**。

**POST `/devices` 请求体（客户 HTML 示例字段）**：`name`（设备名称）、`type`（设备类型）、`location`（**必填**，`space_id`，设备所属空间标识，须为已存在的空间；若不存在则返回 400）；响应含 `device_id` 等，与对外接口说明一致。

## 11. 功能需求：空间管理

空间（space）是设备的物理或逻辑分组单元。空间数据由内部**空间中台**提供，网关负责鉴权、路由与字段映射。集成方通过本节接口管理自身名下的空间，并可在设备注册与查询时按空间进行关联与筛选。

### 11.1 数据来源与映射

| 项目 | 说明 |
|------|------|
| 数据来源 | 内部空间中台 |
| 映射方式 | 网关层调用空间中台能力，对外接口字段与中台字段做映射并在《对外接口说明》中列出 |
| 数据权限 | 所有空间接口均受 appId 租户隔离约束，仅返回当前 appId 下的空间 |

### 11.2 GET `/spaces` — 查询空间列表

**鉴权**：API Key；Scope 需包含读权限（`read` 或 `read_write` 或 `admin`）。

**查询参数**：`limit`、`offset`（分页）

**响应（200）**：

```json
{
  "data": [
    {
      "space_id": "room1",
      "name": "1号会议室",
      "description": "一楼东侧会议室",
      "created_at": "2026-04-20T10:00:00Z"
    }
  ],
  "total": 1
}
```

### 11.3 POST `/spaces` — 创建空间

**鉴权**：API Key；Scope 需为 `admin`。

**请求体**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 是 | 空间名称 |

**响应（201）**：包含 `space_id`、`name`、`created_at` 等。

### 11.4 PUT `/spaces/{space_id}` — 修改空间

**鉴权**：API Key；Scope 需为 `admin`。

**约束**：`space_id` 必须属于当前 appId；否则 404 或 403。

**请求体**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 否 | 空间名称（不传则不修改） |

**响应（200）**：返回更新后的空间信息。

### 11.5 DELETE `/spaces/{space_id}` — 删除空间

**鉴权**：API Key；Scope 需为 `admin`。

**约束**：
- `space_id` 必须属于当前 appId；否则 404 或 403。
- 若该空间下仍存在已注册设备，应拒绝删除并返回 **409 Conflict**，提示先迁移或删除关联设备。

**响应（200）**：如 `{ "status": "deleted" }`。

## 12. 验收要点

- [ ] 使用有效 appId/appSecret 可完成 POST/GET/DELETE/PUT 密钥管理全流程；非法凭证返回 401。
- [ ] 使用只读 Scope 的 Key 无法调用写接口；非 admin Scope 无法调用第 10 节管理接口。
- [ ] 新创建的 api_key 明文仅出现在创建（或轮换）响应中；库中为哈希；列表无明文。
- [ ] 核心读、写接口路径与主要字段与客户规范一致；分页与过滤可用；**`GET /devices` 的 `location`（空间）、`GET …/activity` 的 `from`/`to`、`GET /activity/summary` 的 `period`** 行为符合 §8.2。
- [ ] **第 10 节** Admin 接口（`POST/DELETE /devices`）一期可用；仅 **admin** Scope 可调通。
- [ ] **第 11 节** 空间管理接口（`GET/POST/PUT/DELETE /spaces`）一期可用；查询接口需 `read` 及以上 Scope，写与管理接口需 `admin` Scope。
- [ ] 空间删除时，若该空间下仍有已注册设备，返回 409 Conflict；注册设备时 `location` 引用不存在的 `space_id` 返回 400。
- [ ] 提供《对外接口说明》（含鉴权示例、错误示例、Scope 表、空间管理接口说明、与本文档一致的「无沙箱」说明）。
