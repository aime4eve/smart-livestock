# Commerce 限界上下文设计评审

**评审对象**: Section 1 — Commerce 限界上下文定位与核心模型
**日期**: 2026-05-17

---

## 一、总体评价

骨架设计合理，可以推进。DDD 分层标准，与现有 identity/ranch/iot 的边界清晰，聚合根划分粒度适当。

---

## 二、确认无异议的部分

### 2.1 上下文定位

Commerce 作为独立限界上下文与 identity/ranch/iot 平级，职责边界清晰。目录结构（domain/application/infrastructure/interfaces）符合标准 DDD 六边形架构。

### 2.2 Tenant.type 与 Tenant.billingModel

两个字段放在 Identity/Tenant 上是正确的。它们是注册时确定的路由策略字段，不是计费细节：

- `type` 回答"你是谁"（rancher / reseller / enterprise / developer）
- `billingModel` 回答"怎么收钱"（direct / revenue_share / licensed / api_usage）

两者正交，理论上可组合（如 reseller + licensed 混合）。用 String 而非枚举，已预留扩展空间。

### 2.3 聚合根划分

| 聚合根 | 职责 | 评价 |
|--------|------|------|
| Subscription | 订阅生命周期管理 | 合理，核心聚合根 |
| Contract | 合同与分润条款 | 合理，与 Subscription 解耦 |
| RevenuePeriod | 收入结算周期 | 合理 |
| SubscriptionService | Licensed 模式的服务定义 | 合理，独立于 Subscription |

### 2.4 TenantPhase 状态机

SAMPLE → BATCH 的简化方向可接受。

## 三、建议调整的部分

### 3.1 QuotaResult 分层位置

**现状**: QuotaResult 放在 `domain/model/`
**问题**: QuotaResult 是配额校验的计算结果，不是持久化对象
**建议**: 移至 `application/dto/`，domain/model 只保留持久化的实体和值对象

### 3.2 FeatureGate 归属需明确

FeatureGate 作为值对象，当前未明确挂在哪个聚合根下：
- 如果是按 Subscription 级别控制功能开关 → 挂在 Subscription
- 如果是按 SubscriptionService（licensed）定义服务能力 → 挂在 SubscriptionService
- 建议在模型中显式标注归属关系

### 3.3 billingModel 的实际生效位置

Tenant.billingModel 是注册时的默认路由策略，合理。但建议在 Contract 或 Subscription 层面也存储一份 billingModel，作为实际生效的计费模型：

```
Tenant.billingModel → 默认值（路由用）
Contract.billingModel → 实际值（支持混合模式覆盖）
```

这样未来支持 reseller + licensed 混合时，不需要改 Tenant 结构。

### 3.4 TenantPhase 命名建议

SAMPLE / BATCH 的原始语义是"试用/批量"，重新定义为"试用期/正式付费期"后，语义差异较大。两个选择：

- **方案A**: 保留原名，文档中显式重定义（适合团队已熟悉的情况）
- **方案B**: 重命名为 `TRIAL / ACTIVE`，语义更直观

不影响架构，视团队习惯决定。

## 四、建议补充的部分

### 4.1 账单明细模型

骨架中没有 Billing / Invoice 相关模型。RevenuePeriod 覆盖了结算，但如果需要向牧场主展示月度账单明细（设备费 + 平台费 + 超额费），需要明确：

- 账单展示是 RevenuePeriod 的职责范围？
- 还是需要单独的 Invoice / BillingItem 模型？

**建议**: 如果 Q1/Q2 不做账单明细展示，可以在当前骨架中不加入，但预留 application 层的 BillingApplicationService 位置。

### 4.2 配额门控的拦截层级

QuotaInterceptor 放在 shared/security/ 可以工作，但从职责归属看，配额门控是 Commerce 的业务规则而非通用安全机制。

**建议**: 考虑放在 commerce/interfaces/ 下，或 commerce/application/ 下作为应用服务的一部分，而非 shared 层。

## 五、评审结论

| 维度 | 结论 |
|------|------|
| 架构方向 | 正确，可以推进 |

| 建议修改 | FeatureGate 归属明确（3.2）、billingModel 双层存储（3.3）、QuotaInterceptor 位置（4.2） |
| 可延后 | TenantPhase 命名（3.4）、账单模型（4.1） |
