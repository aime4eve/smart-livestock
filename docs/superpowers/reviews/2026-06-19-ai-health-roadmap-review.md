# 评审：AI 健康智能路线图（Roadmap）

> 评审对象：`docs/superpowers/specs/2026-06-19-ai-health-roadmap.md`
> 评审日期：2026-06-19

## 总体评价

这是一份质量较高的战略总账文档。决策链清晰、有防遗忘机制（§1 + §8）、有业界依据（§2）、issue 双向链接到位（§5）、Phase A/B/C 演进路线前向兼容逻辑自洽。下面按维度给出评审意见。

### ✅ 做得好的地方

- **决策链防遗忘**（§1）：9 个决策点都带"核心理由"，未来回看不会丢失上下文。决策 #2（实现纪律）尤其重要——明确"不为模拟器数据加分支"避免了技术债陷阱。
- **§8 关键认知修正**：把历史上踩过的坑（加速度计误判、四层→三层）固化下来，这是很好的"防再犯"机制。
- **演进策略前向兼容**（决策 #3）：Phase A 异常分数 → Phase B 标注种子，主动学习闭环设计合理。
- **issue 双向链接**（§5）：6 个 issue 都有阶段归属和优先级，阻塞关系（#55/#56 阻塞 Phase B）清晰。
- **Phase A 待定项**（§6）：诚实标注未决问题，没有假装都已解决。

### ⚠️ 需要改进的问题

**1. 决策 #2 存在内在张力，需澄清边界**

决策 #2 说"按真实数据标准实现，不为数据来源加分支"，但 §6 又把"Phase A 评估方式：无标注如何验证"列为待定。这里有一个未明说的风险：**模拟器数据的统计分布可能与真实数据差异很大**（模拟器往往生成平滑/理想化的时序），导致按真实标准实现的算法在模拟器数据上表现"过于好"或"过于差"，给出虚假信心。

建议在 §1 决策 #2 或 §6 显式记录这个风险，并说明：Phase A 的"按真实数据标准实现"指的是**代码路径不分支**，但**评估结论不等于真实数据上的结论**——必须等 #55 接入后重新评估。

**2. §3 架构图中 orchestration/engine 的"透传"价值未论证**

Phase A 明确说 orchestration 和 engine 都是"透传"（orchestration: 请求→TaskPlan→engine；engine: TaskPlan→调 capability）。如果两层都只是透传，**为什么 Phase A 要建三层而不是一层直接调 capability？** 文档说"三层为框架目标"，但缺少对"Phase A 就建空壳层是否过度设计"的反驳。

建议补一句：Phase A 建立透传层是为了**固定接口契约**（orchestration 端点 URL、engine→capability 调用协议），使 Phase B/C 填充时不破坏调用方。否则按 §AGENTS.md 第 2 条"简单优先"，一层更合适。现在的表述容易被质疑为 YAGNI 违反。

**3. §4 Phase B/C 的"前置"与路线图时间线缺失**

Phase B 前置 #55/#56，Phase C 前置"真实数据攒够（数百头牛×数月）"。但文档没有给出任何**时间估计或里程碑触发条件**。"数百头牛×数月"是多久？是 3 个月还是 12 个月？这直接影响 Phase A 异常分数持久化表（§6）需要预留多久的容量。

建议在 §4 每个 Phase 补一行"触发条件/退出条件"，至少定性说明。否则 Phase B/C 会变成永远"等数据"的无期限 backlog。

**4. 决策 #5 的 router 路由阈值未定义**

"按 per-individual 有效样本量 N_eff 分档路由"，但**分档阈值（N_eff 的临界值）在文档中完全没有**。决策 #5 说"小样本用稳健统计，大样本升级 iForest"，多大算大？这是 Phase A design doc 的事（§7 待写），但作为战略决策，#5 至少应该给出**量级方向**（如 N_eff < 50 用 Mahalanobis，> 200 用 iForest），否则"按规模自动选"就是空话。

建议：要么在 §6 待定项里显式列出"router 分档阈值"作为 design 待决项（目前 §6 漏了这条），要么在决策 #5 给一个占位区间。

**5. §6 待定项遗漏了 router 分档阈值**

承接上一条，§6 列了 6 项待定，但**没有"router 按 N_eff 的分档阈值"**。这是 Phase A 的核心算法决策，不应遗漏。建议补入。

**6. 降级链的冷启动场景描述不够具体**

决策 #6 说降级链是"通用可用性设计（新牧场冷启动、模型未就绪）"，但**冷启动时 router 该路由到哪一档？** 新牧场 N_eff 极小（可能 < 10 个样本），Mahalanobis 也需要足够样本估协方差。文档没有说明 N_eff 极小（< 最小阈值）时的兜底——是降级到纯规则引擎（L1 的规则档）？还是返回"数据不足"？

建议在 §6 待定项补一条：**N_eff 极小时的兜底策略**。这关系到 §3 router 路由维度①"能力可用性 L3→L2→L1 降级"在 L1 内部是否还有"算法→规则"的再降级。

### 📝 小问题

- §2 的调研链接用了 `developer.aliyun.com` 转载 Google 白皮书，建议换官方源（`developers.google.com` 或 arXiv），转载链接可能失效。
- §5 #59 标注"探索 backlog"，但优先级列写 "low"——backlog 和 low 优先级语义重复，可简化。
- §3 ASCII 图里 `Java 后端 ──HTTP──→ orchestration 端点` 只画了入向，没画出向（ai-platform 如何回调 Java 后端写异常分数？还是 Java 轮询？）。Phase A 异常分数持久化（§4 衔接）的写入方应该在图里体现。
