> **评审日期**: 2026-05-18

## 一、评审结论

结论先行：**当前项目最适合继续沿用"DDD 限界上下文 + 模块化单体 + 局部事件驱动 + 前后端分离"的总体架构**，不建议现阶段全面拆分为微服务。

当前后端代码已经具备正确骨架：

- 已按 `identity / ranch / iot / shared` 进行业务边界拆分
- 已形成 `domain / application / infrastructure / interfaces` 的基础分层
- 已具备多租户、Farm Scope、JWT、Open API、事件桥接等关键横切能力

这说明项目的**方向是对的**。后续工作的重点不应是"推倒重来换架构"，而应是：

1. **收紧边界**，防止 `shared/` 和跨上下文调用继续扩张
2. **统一目录和分层约束**，降低 Phase 2 之后的维护成本
3. **把高吞吐链路与交易型主链路分开演进**
4. **用演进式服务化替代过早微服务**

## 二、优先级问题汇总

### P1 — 架构层面需要尽快收口

#### A1. `shared/` 已开始承担过多职责，存在"公共大杂烩"风险

当前 `shared/` 下同时放置了：

- `domain/`
- `common/`
- `security/`
- `cache/`
- `messaging/`
- `scope/`
- `tenant/`
- `WebMvcConfig`

这在 Phase 1 是高性价比做法，但如果继续扩展，`shared/` 很容易退化成所有上下文都能依赖的"万能包"，最终削弱限界上下文边界。

**风险**:
- 任何上下文都可直接依赖 `shared/` 中的实现细节
- 后续新增 `commerce/`、`health/` 后，平台能力和业务能力容易混在一起
- 将来若要拆分 `iot ingestion` 或 `analytics query`，会发现依赖树过于交叉

**建议**:
- `shared/domain/` 仅保留真正可共享的领域基类，如 `AggregateRoot`、`DomainEvent`、`Entity`
- 将 `security / tenant / scope / requestId / exception / web config` 归入 `platform/` 或 `bootstrap/`
- 将 `cache / messaging` 明确为基础设施能力，不再以"共享领域能力"暴露

**v2 修正 — `ApiException` 与 `ErrorCode` 的归属**：

> 初版建议将 `ApiException` 和 `ErrorCode` 归入 `platform/web/`。经代码审查发现，**domain 层有 5 个聚合根直接依赖这两个类**：
> - `identity/domain/model/User.java`
> - `identity/domain/model/Tenant.java`
> - `ranch/domain/model/Alert.java`
> - `iot/domain/model/Device.java`
> - `iot/domain/model/Installation.java`
>
> 这说明 `ApiException` 不是 Web 层概念，而是领域层表达业务约束违反的机制。如果放到 `platform/web/`，按照 T1 的 ArchUnit 规则（`domain` 不依赖 `platform/web`），所有领域模型将编译失败。
>
> **修正方案**：将 `ApiException` + `ErrorCode` 保留在 `shared-kernel/domain/`，作为跨所有层的业务异常抽象。或者更精确地拆分：
> - `DomainException`（放在 `shared-kernel/domain/`）— 领域层使用，表达业务规则违反
> - `ApiException`（放在 `platform/web/`）— Web 适配层使用，负责 HTTP 状态码映射
> - 领域模型抛 `DomainException`，`GlobalExceptionHandler` 负责将其转换为 HTTP 响应

建议目标形态：

```text
com.smartlivestock
├── identity/
├── ranch/
├── iot/
├── commerce/
├── health/
├── shared-kernel/
│   └── domain/
│       ├── AggregateRoot.java
│       ├── DomainEvent.java
│       ├── Entity.java
│       ├── DomainException.java    ← v2 新增
│       └── ErrorCode.java          ← v2 修正：从 platform/web 归入
└── platform/
    ├── security/
    ├── tenant/
    ├── web/
    │   ├── ApiException.java       ← v2 修正：仅 Web 适配层使用
    │   ├── GlobalExceptionHandler.java
    │   ├── ApiResponse.java
    │   └── WebMvcConfig.java
    ├── observability/
    ├── messaging/
    └── cache/
```

#### A2. `application/` 层目录风格不统一，后续可维护性会下降

当前各上下文的应用层风格不完全一致：

- `identity/application/` 直接放 `AuthApplicationService.java`
- `iot/application/` 直接放 `DeviceApplicationService.java`，另有 `service/GpsSimulator.java`
- `ranch/application/` 直接放 `AlertApplicationService.java`

这类不一致在当前文件量下问题不大，但进入 Phase 2 后，会明显影响：

- 新成员理解成本
- 用例类检索效率
- 命令、查询、作业、门控、编排逻辑的分层清晰度

**建议**: 尽快统一所有上下文的应用层结构。

推荐目录：

```text
{context}/application/
├── command/
├── query/
├── dto/
├── service/
├── job/
└── assembler/
```

其中：
- `service/` 只放用例编排类
- `job/` 放定时任务、过期处理、批处理作业
- `query/` 放只读模型查询对象
- `assembler/` 放 DTO 映射，避免 controller/application 双向污染

#### A3. IoT 遥测链路仍嵌在主业务单体中，短期合理，中期必须预留分离位

当前 GPS 更新链路是：

```text
Open/API 写入 -> IoT Application -> GpsLogUpdatedEvent -> Ranch EventHandler -> FenceBreach -> Alert
```

Phase 1 这样做完全合理，且工程成本最低。但这条链路与"租户、用户、订阅、合同"相比，具备完全不同的系统特征：

- 写入频率高
- 对吞吐和削峰更敏感
- 更依赖异步处理
- 更容易引入时序数据库、流式处理、消息堆积、幂等控制等问题

**风险**:
- 将高频遥测流量与低频交易型接口耦合在同一应用进程
- 将来真实设备接入后，主单体会同时承担"核心业务事务 + 数据摄取 + 告警触发"
- 性能调优目标冲突，难以兼顾

**建议**:
- 当前继续保留在单体内
- 但立即在概念上划出两层：
  - `iot device management`：设备台账、许可证、安装关系，保留在主业务单体
  - `iot ingestion/telemetry`：设备上报、轨迹写入、异步事件，作为未来首个可拆分能力

#### A4. Query/Read Model 还未完全独立，后续 Dashboard/Map 容易反向侵入领域模型

当前 `DashboardController`、`MapController` 位于 `ranch/interfaces/` 下，这在 MVP 阶段可以接受，但长期看存在一个隐患：

- 写模型和读模型会逐步缠绕
- 领域对象被迫承载展示型字段和聚合查询职责
- 复杂查询容易一路从 controller 穿透到多个 repository

**建议**:
- 继续保留单体，但逻辑上引入 `query` 概念
- 将 Dashboard、Map、Stats 归入独立的只读模型层
- 即便不单独建 `analytics/` 上下文，也建议先建：

```text
ranch/query/
iot/query/
analytics/query/
```

目标原则：**写模型负责规则，读模型负责拼装**。

#### A5. 平台横切能力已有雏形，但观测性仍不是一等公民

当前已经有：

- `RequestIdFilter`
- `TenantContext`
- `FarmScopeInterceptor`
- JWT 认证

说明项目已经在向"平台级后端"演进。但对于 IoT + 多租户系统，真正困难的不是 CRUD，而是排障与归因。

**当前缺口**:
- 缺少统一的业务审计抽象
- 缺少明确的事件追踪链路定义
- 缺少针对设备接入、告警命中、跨上下文事件消费的指标设计

**建议**:
- 在 `platform/observability/` 下集中放：
  - requestId / traceId
  - 结构化日志
  - 审计日志适配层
  - 关键业务指标埋点
- 把"观测性"当作架构组成部分，而不是运维补丁

### P1.5 — v2 新增：跨上下文直接依赖已构成耦合风险

> 初版将跨上下文依赖的解耦（原 T8）列为 P2 最低优先级。经代码审查发现，当前已有 **6 处跨上下文直接 import**，且包含严重的层级越界。将其提升至 P1.5。

**已确认的跨上下文直接依赖**：

| 源 | 目标 | 严重程度 | 说明 |
|---|---|---|---|
| `ranch/infrastructure/event/GpsLogEventHandler` | `iot.domain.event.GpsLogUpdatedEvent` | 中 | Ranch 依赖 IoT 领域事件类 |
| `ranch/infrastructure/event/GpsLogEventHandler` | `iot.domain.model.Installation` | **高** | Ranch 直接引用 IoT 领域模型 |
| `ranch/infrastructure/event/GpsLogEventHandler` | `iot.domain.repository.InstallationRepository` | **高** | Ranch 直接调用 IoT Repository（层级越界） |
| `ranch/interfaces/DashboardController` | `iot.application.DeviceApplicationService` | 中 | Controller 跨上下文调用应用服务 |
| `iot/infrastructure/event/SpringEventPublisher` | `ranch.domain.event.AlertStatusChangedEvent` | 中 | IoT 依赖 Ranch 领域事件类 |
| `iot/infrastructure/event/SpringEventPublisher` | `ranch.domain.event.FenceBreachDetectedEvent` | 中 | IoT 依赖 Ranch 领域事件类 |

**最严重的问题**：`GpsLogEventHandler` 直接调用了 IoT 的 `InstallationRepository`——Ranch 的基础设施层穿透了上下文边界，直接操作了另一个上下文的数据访问对象。这不是通过事件或 port 间接访问，而是硬耦合。

**v2 修正 — SpringEventPublisher 职责错位**：

> 当前 `iot/infrastructure/event/SpringEventPublisher.java` 放在 IoT 上下文下，但它同时转发 Ranch 的事件（`FenceBreachDetectedEvent`、`AlertStatusChangedEvent`）。IoT 包下的类不应了解 Ranch 的事件类型。
>
> **修正方案**：事件桥接是平台级能力，不属于任何业务上下文。将 `SpringEventPublisher` 迁入 `platform/messaging/`，每个上下文可有自己的 EventPublisher（只发布本上下文事件），统一桥接到 MQ 的工作归 `platform/messaging/`。

**修正策略**：

1. **事件类共享**：将跨上下文消费的事件类统一放在 `shared-kernel/domain/event/`，或各上下文定义同名 DTO 通过 Spring Event 类型名匹配解耦
2. **Repository 越界**：`GpsLogEventHandler` 中的 `InstallationRepository` 调用替换为 IoT 上下文提供的 port/interface（如 `DeviceInstallationQueryPort`）
3. **Controller 跨上下文**：`DashboardController` 对 `DeviceApplicationService` 的调用通过 query service 或 facade 间接访问
4. **EventPublisher 归位**：从 `iot/infrastructure/event/` 迁入 `platform/messaging/`

### P2 — 中期建议优化

#### A6. Open API、App API、Admin API 的分面设计正确，应继续强化而非收敛成一套"大接口"

当前三套 API 分面是正确方向：

- App API 面向牧场业务端
- Admin API 面向平台管理
- Open API 面向外部设备/合作方

**建议**:
- 在目录结构上明确三者边界
- controller 包按 `app / admin / open` 拆分
- 权限、限流、审计、幂等策略按入口面分别设计

推荐：

```text
{context}/interfaces/
├── app/
├── admin/
└── open/
```

#### A7. 当前架构适合模块化单体，不适合立即升级为全量微服务

不建议现在全面微服务化，原因不是微服务不好，而是现阶段收益不高：

- 团队主要复杂度仍在业务建模，不在服务治理
- 跨上下文规则仍在快速变化
- 真实 IoT 吞吐场景尚未全面落地
- 如果现在拆，主要得到的是运维复杂度，而不是业务收益

**结论**: 先做"强模块边界的单体"，后做"按吞吐和组织边界拆服务"。

### 3.2 目标架构原则

1. **主业务保持模块化单体**
   - Identity、Ranch、IoT 管理、Commerce 仍在同一部署单元
   - 保留本地事务和开发效率

2. **遥测链路单独视为可拆分能力**
   - GPS、温度、蠕动等上报链路不与租户、合同、权限链路混为一谈

3. **事件驱动只用于"衍生动作"**
   - 核心事务同步完成
   - 告警生成、统计聚合、通知、异步检查使用事件桥接

4. **读写职责分离但不强制独立部署**
   - 先逻辑拆分 read model
   - 是否独立服务由数据量和查询复杂度决定

5. **上下文间只通过 shared-kernel 抽象或 port 通信**（v2 新增）
   - 禁止一个上下文直接 import 另一个上下文的 domain/repository
   - 跨上下文消费的事件类统一放在 `shared-kernel/domain/event/`
   - 跨上下文的数据查询通过 port/interface 解耦

### 4.2 目录调整原则

#### 原则 1: 只共享抽象，不共享业务实现

- `shared-kernel` 只放领域抽象和跨上下文共享的事件 DTO
- 任何业务规则不进入共享层

#### 原则 2: 横切关注点不伪装成业务公共能力

以下内容建议逐步迁出 `shared/` 到 `platform/`：

- `SecurityConfig`
- `JwtAuthenticationFilter`
- `RequestIdFilter`
- `FarmScopeInterceptor`
- `GlobalExceptionHandler`
- `ApiKeyAuthService`
- `WebMvcConfig`

#### 原则 3: Query 与 Command 在目录上显式分开

尤其是以下场景应优先使用 read model：

- Dashboard 汇总
- 地图聚合数据
- 告警统计
- 设备在线率
- 跨上下文运营报表

#### 原则 4: 未来可拆分能力要先在包结构上独立

例如 IoT：

```text
iot/
├── device-management/
└── telemetry/
```

即便仍在单体内，也先把"设备台账"和"遥测摄取"分开。

#### 原则 5: 上下文间禁止直接依赖（v2 新增）

- 一个上下文的任何层都不得直接 import 另一个上下文的 `domain.model`、`domain.repository` 或 `application` 类
- 跨上下文事件消费通过 `shared-kernel/domain/event/` 中的共享事件类解耦
- 跨上下文数据查询通过 port/interface（定义在消费方，实现在提供方）解耦
- 事件桥接（Spring Event → RocketMQ）归 `platform/messaging/`

## 五、目标目录改造方案

本节回答两个问题：

1. **当前目录结构具体要怎么改**
2. **哪些目录需要先改名，哪些目录只需先逻辑收口，不必立刻移动文件**

原则是：**先建立目标骨架，再迁移实现；先收边界，再调命名；优先低风险搬迁，避免一口气大重构。**

### 5.1 改造目标

目标不是"把包名改漂亮"，而是解决当前 5 类真实问题：

- `shared/` 职责过宽，平台能力与共享领域抽象混放
- `interfaces/` 分面不统一，`app / admin / open` 边界不够清晰
- `application/` 层目录风格不一致，后续 Phase 2 文件量会上升
- Query 读模型与写模型未显式分离，Dashboard/Map/Stats 迟早缠绕
- **跨上下文直接依赖已形成硬耦合，影响未来拆分**（v2 新增）

### 5.2 改造边界

#### 5.2.1 本轮改造必须做到的边界

- 共享领域抽象与平台横切能力分离
- 各上下文的 `application/` 结构统一
- 各上下文的 `interfaces/` 结构统一
- Query 代码开始有独立归属
- IoT 内部显式区分 `device-management` 与 `telemetry`

#### 5.3.6 跨上下文依赖解耦方案（v2 新增）

**事件类解耦**：

将跨上下文消费的事件从各上下文的 `domain/event/` 迁入 `shared-kernel/domain/event/`：

| 事件 | 当前位置 | 目标位置 | 消费方 |
|---|---|---|---|
| `GpsLogUpdatedEvent` | `iot/domain/event/` | `shared-kernel/domain/event/` | Ranch EventHandler |
| `FenceBreachDetectedEvent` | `ranch/domain/event/` | `shared-kernel/domain/event/` | IoT EventPublisher |
| `AlertStatusChangedEvent` | `ranch/domain/event/` | `shared-kernel/domain/event/` | IoT EventPublisher |

仅在单一上下文内消费的事件保留在各上下文的 `domain/event/` 下。

**Repository 越界解耦**：

`GpsLogEventHandler` 中的 `InstallationRepository` 调用需要替换为 port：

```text
iot/domain/repository/
├── InstallationRepository.java          (IoT 内部使用)
└── port/
    └── DeviceInstallationQueryPort.java  (v2 新增：供跨上下文查询)

ranch/infrastructure/
└── external/
    └── IotDeviceInstallationQueryAdapter.java  (v2 新增：实现 port，委托 IoT Repository)
```

`DeviceInstallationQueryPort` 定义在 IoT 的 `domain/repository/port/` 下，只暴露查询所需的最小接口（`findActiveDeviceId → livestockId`），不暴露 Installation 实体。`GpsLogEventHandler` 改为依赖 `DeviceInstallationQueryPort` 而非 `InstallationRepository`。

**DashboardController 跨上下文调用解耦**：

`DashboardController` 当前直接调用 `iot.application.DeviceApplicationService`。通过 query service 解耦：

```text
ranch/application/query/
└── DashboardQueryService.java  (聚合 ranch + iot 数据，通过 port 查询 IoT)
```

### 5.4 推荐改造顺序

按风险从低到高排序：

1. 新建目标目录，不移动代码
2. **解耦跨上下文依赖（v2 提升）**
3. 调整 `interfaces/` 目录
4. 调整 `application/` 目录
5. 拆 `shared/` 为 `shared-kernel + platform`
6. 新增 `query/` 目录并迁移 Dashboard/Map 查询逻辑
7. 在 IoT 内部引入 `telemetry` 概念层
8. 最后再考虑物理迁移到 `analytics/`

### 5.5 改造验收标准

目录改造完成后，应满足以下可检查标准：

- 任一业务上下文都不再直接依赖 `platform` 之外的其他上下文实现类
- **任一上下文都不直接 import 另一上下文的 domain/repository/application 类**（v2 新增）
- `shared-kernel` 内不出现 Web、Security、Redis、MQ 相关实现
- 所有 controller 都能在 `app / admin / open` 三类入口中找到明确归属
- 所有 application 用例类都位于 `application/service/`
- 所有复杂读查询都位于 `query/`，而不是 controller 直接拼 repository
- IoT 遥测代码可在目录上被单独识别
- **事件桥接组件（SpringEventBridge）位于 `platform/messaging/`**（v2 新增）

## 七、持久化技术栈补充评审

本节补充回答三个问题：

1. **这个项目的持久化技术应该怎么选**
2. **哪些包继续使用 JPA/Hibernate，哪些包应改用 jOOQ**
3. **如何从当前"以 JPA 为主"的实现平滑迁移到 `JPA + jOOQ` 混合栈**

评审结论先行：

- **核心交易域继续使用 `JPA + Hibernate`**
- **复杂查询和读模型聚合优先使用 `jOOQ`**
- **高频遥测写入不建议继续扩大 JPA 使用面，应优先使用 `jOOQ batch`**

换句话说，本项目更适合的是**混合持久化栈**，而不是"全站统一 ORM"。

### 7.1 持久化技术选型矩阵

| 场景 | 推荐技术 | 是否推荐 JPA/Hibernate | 是否推荐 jOOQ | 说明 |
|---|---|---|---|---|
| 聚合根写模型 CRUD | JPA/Hibernate | 是 | 可选 | 适合聚合生命周期、事务边界清晰的核心业务域 |
| 复杂列表筛选 | jOOQ | 一般 | 是 | SQL 可控、条件组合清晰，优于 Criteria/JPA Specification |
| Dashboard 聚合统计 | jOOQ | 否 | 是 | 聚合、分组、窗口函数、CTE 更自然 |
| 地图/轨迹查询 | jOOQ | 否 | 是 | 更适合 PostGIS、空间函数和复杂联查 |
| 高频批量写入遥测数据 | jOOQ batch | 否 | 是 | 避免实体管理、脏检查、flush 成本 |
| 简单后台管理查询 | JPA/Hibernate | 是 | 可选 | 查询简单、分页简单时继续 JPA 成本最低 |
| 跨上下文读模型拼装 | jOOQ | 一般 | 是 | 读模型天然更适合 SQL-first |
| 定时报表/对账聚合 | jOOQ | 一般 | 是 | 结果导向查询、规则计算、SQL 优势明显 |
| 时序数据起步阶段 | PostgreSQL + jOOQ | 否 | 是 | 先利用 PostgreSQL 分区和 batch 写入 |
| 时序数据扩容阶段 | TimescaleDB + jOOQ | 否 | 是 | 当数据量、保留期和趋势分析压力上来后再升级 |

### 7.2 为什么不建议"全站继续只用 JPA/Hibernate"

`JPA + Hibernate` 在这个项目里不是错误选择，但它不是全局最优选择。

它最适合：

- `Tenant / User / Farm`
- `Livestock / Fence / Alert`
- `Subscription / Contract / RevenuePeriod`
- `Device / DeviceLicense / Installation`

因为这些对象具备：

- 明确的聚合边界
- 稳定的生命周期
- 以业务规则为主，而不是以 SQL 技巧为主
- 需要良好的事务一致性

但它不适合继续覆盖以下场景：

- Dashboard 聚合
- 地图聚合与轨迹历史
- 多表复杂筛选
- 遥测高频写入
- 未来温度/蠕动/加速度等时序查询

对这些场景，如果强行继续用 JPA，通常会出现：

- repository 方法膨胀
- JPQL/Specification 可读性下降
- 对 PostgreSQL 特性利用不足
- 批量写入性能差
- controller/query 层为了拿数据而反复穿透多个仓储

### 7.3 包级持久化技术落位建议

原则：

- **写模型包**继续使用 JPA/Hibernate
- **查询包**优先使用 jOOQ
- **遥测写入链路**优先使用 jOOQ batch

#### 7.3.1 继续使用 JPA/Hibernate 的包

| 包 | 推荐技术 | 原因 |
|---|---|---|
| `identity/domain/*` + `identity/infrastructure/persistence/*` | JPA/Hibernate | 租户、用户、牧场是典型事务型聚合 |
| `ranch/domain/*` + `ranch/infrastructure/persistence/*` | JPA/Hibernate | 牲畜、围栏、告警适合聚合根建模 |
| `iot` 中 `Device / DeviceLicense / Installation` 相关持久化 | JPA/Hibernate | 设备台账、许可证、安装记录是事务型主数据 |
| `commerce` 未来 `Subscription / Contract / RevenuePeriod` | JPA/Hibernate | 订阅、合同、结算周期需要聚合规则和事务边界 |
| `health` 未来若存在评估聚合快照表 | JPA/Hibernate | 如果是"评估结果快照"而非原始时序，可继续用 JPA |

#### 7.3.2 优先改用 jOOQ 的包

| 包 | 推荐技术 | 原因 |
|---|---|---|
| `analytics/query/*` | jOOQ | 跨上下文读模型、聚合和报表天然适合 SQL-first |
| `ranch/query/*` | jOOQ | 牲畜增强列表、告警统计、围栏关联视图 |
| `iot/query/*` | jOOQ | 设备在线率、安装概览、轨迹查询、时序过滤 |
| `iot/telemetry/query/*` | jOOQ | 时序查询、轨迹范围查询、窗口聚合 |
| `iot/telemetry/ingestion/*` | jOOQ batch | 高频批量写入优先 |
| `commerce/query/*` | jOOQ | 收入汇总、分润对账、订阅运营报表 |

#### 7.3.3 当前代码的直接映射建议

| 当前包 / 文件 | 现状 | 建议 |
|---|---|---|
| `identity/infrastructure/persistence/*` | JPA | 保持 JPA |
| `ranch/infrastructure/persistence/*` | JPA | 保持 JPA |
| `iot/infrastructure/persistence/Device*.java` | JPA | 保持 JPA |
| `iot/infrastructure/persistence/Installation*.java` | JPA | 保持 JPA |
| `iot/infrastructure/persistence/GpsLog*.java` | JPA | 中期迁出为 jOOQ 写入/查询 |
| `ranch/interfaces/DashboardController.java` | controller 直连应用层 | 后续对接 `analytics/query` 的 jOOQ 查询服务 |
| `ranch/interfaces/MapController.java` | controller 直连应用层 | 后续对接 `analytics/query` 或 `iot/query` 的 jOOQ 查询服务 |
| 未来 `analytics/query/*` | 尚未建立 | 新建后直接以 jOOQ 为默认技术 |

### 7.4 推荐的混合栈形态

推荐目标形态如下：

```text
写模型（事务域）
  domain -> application/service -> repository(port) -> JPA/Hibernate(adapter)

读模型（查询域）
  query service -> jOOQ DSL -> PostgreSQL

遥测写入
  ingestion service -> jOOQ batch -> PostgreSQL/TimescaleDB
```

换成代码组织可以理解为：

```text
identity/
├── application/service/        -> JPA
└── infrastructure/persistence/ -> JPA

ranch/
├── application/service/        -> JPA
├── query/                      -> jOOQ
└── infrastructure/persistence/ -> JPA

iot/
├── application/service/        -> JPA (device management)
├── query/                      -> jOOQ
├── telemetry/ingestion/        -> jOOQ batch
└── infrastructure/persistence/ -> JPA + jOOQ 并存
```

### 7.5 从当前代码迁移到 `JPA + jOOQ` 混合栈的实施方案

迁移原则：

- **不推翻现有 JPA**
- **先加 jOOQ，再迁移查询**
- **先迁读，再迁高频写**
- **单次提交只切一类能力**

#### 阶段 A: 引入 jOOQ 基础设施，但不替换任何现有实现

**目标**: 让项目具备使用 jOOQ 的能力，但业务行为不变。

**动作**:

1. 在 `smart-livestock-server` 中引入 jOOQ 依赖与代码生成插件
2. 生成 PostgreSQL schema 对应的 DSL 代码
3. 新增 `platform/persistence/jooq/` 或同类基础设施目录
4. 建立统一的 `DSLContext` 配置
5. 保持现有 JPA repository 全部不动

**验收**:

- 项目可正常编译
- jOOQ 代码生成成功
- 现有测试无回归

#### 阶段 B: 为 Query 层创建独立技术通道

**目标**: 不再让复杂查询挤进 JPA repository。

**动作**:

1. 新建 `ranch/query/`、`iot/query/`、`analytics/query/`
2. 为 Query 层建立独立服务，例如：
   - `DashboardQueryService`
   - `MapQueryService`
   - `DeviceOverviewQueryService`
3. Query service 默认通过 jOOQ 访问数据库
4. controller 通过 query service 获取读模型，不再直接拼装 repository

**首批适合迁移的查询**:

- Dashboard 指标汇总
- 地图页牲畜位置与围栏聚合
- 告警统计列表
- 设备在线率和安装概览

**验收**:

- 首批读模型已不依赖 JPA repository 组合查询
- controller 中的聚合逻辑显著减少

#### 阶段 C: 将 `GpsLog` 读写从 JPA 主链路中剥离

**目标**: 避免遥测链路继续扩大 JPA 使用范围。

**动作**:

1. 保留 `Device / DeviceLicense / Installation` 使用 JPA
2. 为 `GpsLog` 新建：
   - `iot/telemetry/ingestion/`
   - `iot/telemetry/query/`
3. GPS 写入改为 jOOQ batch 或单条 SQL
4. GPS 历史查询、范围查询改为 jOOQ
5. 原 JPA `GpsLogRepository` 逐步降级，仅用于过渡或删除

**验收**:

- GPS 写入和历史查询不再依赖 JPA entity 生命周期
- `GpsLogJpaEntity` 不再成为新增时序能力的默认模板

#### 阶段 D: Commerce 与 Analytics 直接采用混合栈

**目标**: 不在新上下文里重复"所有事情都走 JPA"的路径依赖。

**动作**:

1. Commerce 写模型继续 JPA
2. Commerce 报表、运营视图、分润汇总直接使用 jOOQ
3. Analytics 从第一天起就使用 query 包 + jOOQ
4. Health 原始时序/趋势查询优先 jOOQ

**验收**:

- 新增上下文天然区分写模型与读模型
- 不再出现"统计类需求先往 JPA repository 塞方法"的习惯

### 7.6 迁移任务卡建议

如果要把本节内容继续落成开发任务，建议增加以下任务卡：

#### T8. 引入 jOOQ 基础设施（原 T9 重编号）

**目标**: 建立混合栈基础能力。

**范围**:
- Gradle 依赖
- jOOQ 代码生成

**验收**:

#### T9. 首批 Query 场景迁移到 jOOQ（原 T10 重编号）

**目标**: 先把最适合 SQL-first 的读模型从 JPA 迁出。

**范围**:
- Dashboard
- Map
- 告警统计
- 设备概览

**验收**:
- 首批读模型 controller 不再拼装多个 JPA repository

#### T10. `GpsLog` 遥测链路迁移到 jOOQ（原 T11 重编号）

**目标**: 为高频写入和时序查询建立正确的技术轨道。

**范围**:
- GPS 写入
- GPS 历史查询
- 批量写入路径

**验收**:
- `GpsLog` 从"JPA 默认场景"切换为"jOOQ 默认场景"

### 7.7 最终建议

对这个项目，最佳选择不是"抛弃 JPA/Hibernate"，而是：

- **核心事务域保留 JPA/Hibernate**
- **复杂查询与读模型引入 jOOQ**
- **遥测高频写入优先走 jOOQ batch**

这条路线兼顾了：

- 当前代码成本
- 团队可接受的迁移复杂度
- DDD 写模型的一致性
- PostgreSQL 与时序/空间查询能力的发挥空间

## 八、为什么这是最适合当前项目的架构

### 8.1 业务复杂度决定更适合 DDD，而不是简单三层 CRUD

这个项目的核心难点是：

- 多租户
- 多角色
- 牧场作用域
- 设备与牲畜安装关系
- 围栏越界告警
- 订阅和配额门控
- 后续健康分析与商业计费

这类系统更适合按业务边界建模，而不是按 `controller/service/repository` 平铺。

### 8.2 工程阶段决定更适合模块化单体，而不是全面微服务

当前阶段更需要：

- 稳定边界
- 降低交付成本
- 提高本地调试效率
- 保持事务和联调简单

模块化单体正好满足这些目标。

### 8.3 技术特征决定 IoT 链路应该"优先独立概念"，但不必立即独立部署

IoT 与交易管理的系统特征差异太大，所以必须预留拆分位；但当前如果立即拆分，又会增加：

- 消息一致性成本
- 本地开发成本
- 部署和排障成本

因此最佳策略是：**先包级隔离，再进程级拆分**。

### 阶段 1: 现在到 Phase 2 前半段

目标：**把模块化单体做扎实**

### 阶段 2: Commerce/Health 落地后

目标：**把读模型和高频链路独立成可运维能力**

建议动作：

1. 将 `iot telemetry` 在包结构和部署结构上准备独立
2. 让告警生成、统计聚合更多通过事件驱动完成

- `telemetry-ingestion-service`

不建议第一个拆的是：

- `identity-service`
- `ranch-service`
- `commerce-service`

因为这些主业务仍强依赖本地事务和协同开发效率。

### 阶段 3: 真实设备规模上来后

目标：**按吞吐和组织边界做有限服务化**

推荐优先级：

1. `Telemetry Ingestion Service`
2. `Alert/Rule Engine`（当规则复杂到独立配置、灰度、回放时）
3. `Analytics Query Service`（当地图和统计查询压到主库时）

仍不建议拆成大量细粒度服务。推荐保持"少量粗粒度服务"。
