# 国际化（i18n）设计规格 — 中英文在线切换

> **日期**：2026-06-11
> **状态**：已评审，待实施
> **范围**：Flutter App + Spring Boot 后端（PC/Angular 与 Mock Server 不做）
> **关联**：基于 `Mobile/CLAUDE.md` 与 `smart-livestock-server/` 现状

---

## 1. 目标与场景

### 1.1 目标

实现中英文**在线切换**（运行时切换语言，无需重启 App、无需重新登录），覆盖：

- Flutter App 的界面文案（UI 静态文本 + 业务枚举显示名）
- Spring Boot 后端的动态消息（错误、校验、业务提示）

### 1.2 范围

| 部分 | 是否做 | 说明 |
|---|---|---|
| Flutter App（`Mobile/mobile_app`） | ✅ | UI 文案 + 枚举显示名 |
| Spring Boot 后端（`smart-livestock-server`） | ✅ | 错误/校验/业务提示消息 |
| PC/Angular 端（`PC/`） | ❌ | 暂不维护 |
| Node Mock Server（`Mobile/backend`） | ❌ | 仅前端框架文案可切换；mock 模式下业务数据保持中文（可接受，出海用 live 模式） |

> **影响说明**：因 Mock Server 不翻译，mock 模式下界面框架文案（按钮/标题）能切英文，但 Mock Server 返回的业务数据（告警类型名、设备名）保持中文，会出现中英混杂。live 模式（真实后端）全链路一致，不受影响。

### 1.3 场景：出海产品

面向海外牧场/客户。硬要求：

- 翻译**专业准确**（人工校对畜牧/IoT 领域术语，不机翻凑数）
- 架构**为后续语言预留扩展**（加一种语言 = 加一套文件，不改代码）

### 1.4 语言覆盖

首发：`zh-CN`（默认）+ `en`。扩展预留：key 命名规范 + 平铺文件结构。

## 2. 整体架构与分工

### 2.1 方案选型：混合分层（方案 C）

经三方案对比（后端完整 i18n / 前端唯一翻译层 / 混合分层），选定**混合分层**：后端负责"它动态产生的消息"，前端负责"静态 UI 文案 + 枚举显示名"。理由：职责清晰、参数化消息后端占位符处理最干净、枚举不动 DB、契合出海产品的可扩展要求。

### 2.2 三类文本翻译归属

| 文本类型 | 例子 | 翻译归属 | 机制 |
|---|---|---|---|
| UI 静态文案 | 按钮"确认"、标题"牧场"、表单标签"手机号" | 前端 | Flutter `gen-l10n` + `.arb` |
| 业务枚举显示名 | `AlertType.FENCE_BREACH` → "围栏越界" | 前端 | 同样走 `.arb`（key = `enum.alertType.fenceBreach`） |
| 后端动态消息 | "已达到上限 5，当前 3"、"手机号不能为空" | 后端 | Spring `MessageSource` + `messages_*.properties` |

**关键**：枚举值（如 `FENCE_BREACH`）在前后端之间传输不变，只有显示名按 locale 翻译。因此**后端枚举与数据库 schema 零改动**。

### 2.3 数据流

```
用户在「我的/设置」切换语言
  → LocaleController(Riverpod) 更新 Locale，写入 shared_preferences
  → MaterialApp.locale 变化 → gen-l10n 用新 .arb 立即重渲染整个 UI（无需重启 App）
  → ApiClient 在后续所有请求附带 Accept-Language: zh-CN / en
  → 后端 LocaleResolver 解析 → MessageSource 返回对应语言的动态消息
  → 枚举值照常传输，前端按当前 locale 查 .arb 显示名渲染
```

### 2.4 关键策略

- **默认语言**：首次启动跟随系统 locale；用户手动切换后持久化，覆盖系统设置。
- **后端 fallback**：收到不识别的 `Accept-Language` → fallback 到 `zh-CN`。
- **语言契约**：标准 HTTP `Accept-Language` header，不引入自定义参数。

## 3. 后端 i18n 设计

### 3.1 i18n 基础设施（Spring 标准方案）

- 引入 `MessageSource`（Spring Boot 自动装配，只需加 properties 文件 + 配置 basename）
- 文件结构：
  ```
  src/main/resources/
    messages.properties        # 默认/fallback（内容同 zh）
    messages_zh.properties     # 中文
    messages_en.properties     # 英文
  ```
- **Locale 解析**：用 Spring 内置 `AcceptHeaderLocaleResolver`（默认从 `Accept-Language` 解析），配置 fallback 到 `zh-CN`。**不写自定义 resolver**。
- **key 命名规范**：`error.{errorCode}.{场景}` 或 `{上下文}.{场景}`，例如：
  - `error.subscription.notFound = 订阅不存在: tenantId={0}` / `Subscription not found: tenantId={0}`
  - `error.quota.limitReached = 已达到上限 {0}，当前: {1}` / `Limit reached {0}, current: {1}`
  - `validation.phone.required = 手机号不能为空` / `Phone number is required`

### 3.2 ApiException 改造（核心）

现状：`ApiException(ErrorCode code, String message)`，message 为中文硬编码。

改造：新增 `messageKey` + `args` 字段，message 语义从"中文文本"变为"i18n key"。

```java
public class ApiException extends RuntimeException {
    private final ErrorCode code;
    private final String messageKey;   // i18n key，如 "error.subscription.notFound"
    private final Object[] args;       // 占位符参数 {0} {1} ...

    public ApiException(ErrorCode code, String messageKey, Object... args) {
        super(messageKey);
        this.code = code;
        this.messageKey = messageKey;
        this.args = args;
    }

    public ErrorCode getCode() { return code; }
    public String getMessageKey() { return messageKey; }
    public Object[] getArgs() { return args; }
}
```

**Before → After**（60 个 throw 点的机械化迁移）：

```java
// Before
throw new ApiException(ErrorCode.RESOURCE_NOT_FOUND, "订阅不存在: tenantId=" + tenantId);
throw new ApiException(ErrorCode.QUOTA_EXCEEDED, "已达到上限 " + limit + "，当前: " + usage);

// After
throw new ApiException(ErrorCode.RESOURCE_NOT_FOUND, "error.subscription.notFound", tenantId);
throw new ApiException(ErrorCode.QUOTA_EXCEEDED, "error.quota.limitReached", limit, usage);
```

对应 `messages_zh.properties` / `messages_en.properties` 各加一行。参数化消息的拼接交给 `MessageSource` 的 `{0}{1}` 占位符。

- **一次性迁移**：60 个文件的 throw 点全部改成 key+args（机械化但量大）。
- **不漏改保障**：旧构造签名变化 → 编译期强制所有调用点改造。
- **DomainException 同构改造**（同为 code + message 机制）。

### 3.3 GlobalExceptionHandler 改造（i18n 收敛点）

`GlobalExceptionHandler` 是唯一把 message 返回给前端的出口。注入 `MessageSource`，在每个 handler 用 `LocaleContextHolder.getLocale()` 取 locale 后翻译 messageKey：

```java
@ExceptionHandler(ApiException.class)
public ResponseEntity<ApiResponse<Void>> handleApiException(ApiException ex) {
    Locale locale = LocaleContextHolder.getLocale();
    String message = messageSource.getMessage(
        ex.getMessageKey(), ex.getArgs(), ex.getMessageKey(), locale);
    // 第 3 个参数 = defaultMessage：找不到 key 时用 key 本身（防 properties 漏配置）
    String requestId = currentRequestId();
    HttpStatus status = mapToHttpStatus(ex.getCode());
    return ResponseEntity.status(status)
        .body(ApiResponse.error(ex.getCode(), message, requestId));
}
```

- **响应契约不变**：仍是 `{ code, message, requestId }`，前端零改动，只是 `message` 内容按 locale 变。
- `DomainException`、`AccessDeniedException`（"权限不足"）、`MethodArgumentNotValidException` 等各 handler 同样走 MessageSource。

### 3.4 Bean Validation 消息 i18n

Controller 校验注解用 `{key}` 引用 properties：

```java
@NotBlank(message = "{validation.phone.required}") String phone
```

`MethodArgumentNotValidException` handler 同样走 MessageSource。全项目 Controller 的校验注解 message 扫一遍改成 key。

### 3.5 日志处理

日志**不依赖 locale**，固定打 `[code] messageKey args`（如 `ApiException QUOTA_EXCEEDED error.quota.limitReached [5,3]`）—— 英文/key 便于运维检索，给前端的翻译版只在响应里。

## 4. 前端 i18n 设计

### 4.1 gen-l10n 接入（Flutter 官方方案）

- `pubspec.yaml` 加依赖 `flutter_localizations` + `intl`，开 `generate: true`，加 `l10n.yaml` 配置（`arb-dir: lib/l10n`，`template-arb-file: app_zh.arb`）
- `.arb` 文件：`lib/l10n/app_zh.arb`（默认/模板）、`lib/l10n/app_en.arb`
- 生成强类型 `AppLocalizations`，通过 `AppLocalizations.of(context)` 访问（key 用点号，getter 为 camelCase，如 `nav.login` → `l10n.navLogin`）
- `DemoApp` 的 `MaterialApp.router` 加 `localizationsDelegates`、`supportedLocales: [Locale('zh'), Locale('en')]`、`locale`（从 Riverpod 取）
- 选 gen-l10n 而非 easy_localization：官方维护、原生支持 ICU 复数/占位符，契合出海扩展需求

### 4.2 Locale 状态管理（Riverpod）

- `LocaleController`（Notifier）：持有 `Locale`，`setLocale()` 更新 + 写 `shared_preferences`
- 暴露 `localeProvider`；`DemoApp` 的 `MaterialApp.locale` watch 它
- 初始 locale：启动读 `shared_preferences`，无值则跟随系统 `PlatformDispatcher.locale`
- **桥接 ApiClient**：locale 变化时调用 `ApiClient.instance.setLocale(...)`，与 `setActiveFarmId` 同模式

### 4.3 "在线切换"机制（无需重启）

```
LocaleController.setLocale(en)
  → localeProvider 更新
  → MaterialApp.locale 变化 → gen-l10n 用 app_en.arb 重建整个 UI（Flutter 标准能力）
  → ApiClient.instance.setLocale('en') → 后续请求带 Accept-Language: en
  → Riverpod 业务状态不重置（不重新登录、不重拉数据）
```

**切换后不需要刷新数据**：枚举显示名是前端翻译的，UI 自动重渲染；后端错误消息只在下次请求时按新语言返回。

### 4.4 语言切换 UI

放在**"我的"页面**的设置区，一个语言选择器。提供三选：

- **中文**（固定 `zh`）
- **English**（固定 `en`）
- **跟随系统**（清除持久化，回到 `PlatformDispatcher.locale`）

切换即时生效 + 持久化。

### 4.5 ApiClient 改造

```dart
String? _locale;  // 'zh-CN' / 'en'
void setLocale(String locale) => _locale = locale;

Future<Map<String, String>> _headers() async {
  final token = await JwtStorage.instance.getAccessToken();
  return {
    'Content-Type': 'application/json',
    if (token != null) 'Authorization': 'Bearer $token',
    if (_locale != null) 'Accept-Language': _locale,
  };
}
```

- 后端返回的 `body['message']` 直接用（方案 C 后端已按 locale 翻译），各类 Exception 的 message 自动随语言变。
- 4 处硬编码 fallback 中文改为 `.arb` key：
  - `'认证失败'`（401）
  - `'服务器异常'`（500 解析失败）
  - `'租户已禁用'`（403 TENANT_DISABLED）
  - `'登录失败'`（login 失败）

### 4.6 UI 文案抽取（工作量主体）

| 范围 | 处理 |
|---|---|
| 39 个 `AppRoute` 第三参数 | 语义从"中文名"→"i18n key"（如 `'登录'` → `'nav.login'`），`main_shell.dart` 等消费处翻译 |
| 26 个功能模块页面 | 所有 `Text('确认')` → `Text(l10n.commonConfirm)`，逐文件抽取 |
| 业务枚举显示名 | 前端枚举值 → `.arb` key 映射，helper 按当前 locale 翻译（如 `enum.alertType.fenceBreach`） |

枚举显示名表覆盖所有面向用户展示的枚举（`AlertType`、`AlertStatus`、`DeviceType`、`DeviceStatus`、`Role`、`SubscriptionTier`、`SubscriptionStatus`、`ContractStatus`、`Severity`、`HealthStatus` 等），清单化逐一建表。

### 4.7 测试同步

60 个测试文件里的中文断言 → 固定 `Locale('zh')` 下断言中文文本（测试默认走中文 locale，保证稳定）。

## 5. 翻译流程与质量保障

### 5.1 翻译来源

**机翻初稿（DeepL 等）+ 人工专业校对**。重点校对畜牧/IoT 领域术语：

| 中文 | 英文（需专业确认） |
|---|---|
| 瘤胃胶囊 | rumen capsule |
| 发情识别 | estrus detection |
| 围栏越界 | fence breach |
| 耳标 / 追踪器 | ear tag / tracker |
| 牧工 | herder / ranch worker |
| 消化（管理） | digestive |
| 疫病防控 | epidemic prevention |

### 5.2 术语表（glossary）

建立中英术语对照表，保证全项目术语一致（同一词不出现两种译法）。术语表作为翻译质量基准，纳入仓库维护。

### 5.3 维护流程

- 开发新增文本时先写中文 key，英文异步补。
- **CI 校验 key 完整性**：`app_zh.arb` 与 `app_en.arb`（及后端 `messages_zh.properties` / `messages_en.properties`）的 key 集合必须完全一致，缺失则构建失败。

## 6. 测试策略

### 6.1 后端

- **单元测试**：`MessageSource.getMessage(key, args, locale)` 翻译正确性（含占位符参数）
- **集成测试**（Testcontainers）：模拟 `Accept-Language: en` / `zh-CN` 请求，断言错误响应 message 按语言变化
- **key 完整性测试**：`messages_zh.properties` 与 `messages_en.properties` 的 key 集合相等（防止漏译）
- 全量 `./gradlew test` 保持绿色

### 6.2 前端

- **编译期保证**：gen-l10n 缺 key 直接编译失败
- **widget 测试**：固定 `Locale('zh')` 断言中文文本
- **切换测试**：切英文后 UI 文本变化 + `ApiClient` header 带正确 `Accept-Language`
- **枚举翻译 helper 单元测试**
- 全量 `flutter test` + `flutter analyze` 保持绿色

## 7. 实施顺序

```
阶段 0  基础设施搭建（先打通链路，零业务迁移）
  后端: MessageSource + 空 properties + GlobalExceptionHandler 改造 + Locale fallback
  前端: gen-l10n 接入 + LocaleController + shared_preferences 持久化
        + ApiClient.setLocale + MaterialApp 配置 + "我的"页切换 UI
  → verify: 切换语言，少量示例 key 能切；后端示例错误按 Accept-Language 变化

阶段 1  后端消息迁移（60 文件 throw → key+args + Bean Validation 注解）
  → verify: ./gradlew test 全绿；curl 带 Accept-Language 验证错误多语言

阶段 2  前端 UI 文案抽取（AppRoute + 26 模块页面 + 枚举显示名表）
  → verify: flutter test 全绿；切换英文后 grep 无中文残留

阶段 3  收尾（英文术语专业校对 + CI key 完整性检查 + 文档更新）
  → verify: 术语表完成；CI 通过；全链路中英切换无遗漏
```

阶段 1 与阶段 2 可并行（前后端独立）。

## 8. 工作量与风险

### 8.1 工作量

- 后端：~60 文件 throw 点迁移 + Bean Validation 注解 + properties 编写
- 前端：~26 功能模块页面文案抽取 + 39 路由 label + 枚举显示名表
- 测试：~60 前端测试文件断言调整 + 后端新增 i18n 测试
- 翻译：英文版全文 + 术语表

量级较大，分阶段推进，阶段 1/2 可并行。

### 8.2 风险与对策

| 风险 | 对策 |
|---|---|
| 前端文案抽取遗漏 | 切换英文后 grep 中文残留扫描兜底 |
| 测试断言中文失效 | 统一固定 `Locale('zh')` |
| 枚举显示名遗漏 | 枚举清单化（20 个后端枚举逐一建表） |
| 翻译术语不一致 | glossary + 人工校对 |
| properties/.arb 漏 key | CI key 完整性检查 |

## 9. 关键决策记录

| 决策点 | 结论 |
|---|---|
| 范围 | App（Flutter）+ 后端（Spring Boot）；PC/Angular 与 Mock Server 不做 |
| 场景 | 出海产品（专业翻译 + 可扩展） |
| 方案选型 | 混合分层（方案 C）：动态消息归后端，静态文案+枚举归前端 |
| ApiException 迁移 | 一次性迁移 60 个 throw 点为 key+args |
| Bean Validation | 纳入 i18n 范围 |
| 语言切换 UI | "我的"页面设置区，三选（中文/English/跟随系统） |
| AppRoute 第三参数 | 语义从"中文名"改为"i18n key" |
| 后端 Locale 解析 | 标准 `Accept-Language` header + `AcceptHeaderLocaleResolver`，fallback `zh-CN` |
| 默认语言 | 跟随系统 locale，手动切换后持久化覆盖 |
| 响应契约 | 不变（`{ code, message, requestId }`） |
