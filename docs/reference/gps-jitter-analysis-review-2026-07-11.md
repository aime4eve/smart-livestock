# GPS 抖动分析审阅意见

- **日期**: 2026-07-11
- **审阅对象**: [gps-jitter-analysis-2026-07-11.md](./gps-jitter-analysis-2026-07-11.md)

## 认可的部分

1. **数据验证充分**：536 条记录、13 个唯一坐标对的统计很直观，能清楚说明"离散锁定位置反复跳变"而非均匀随机噪声的结论。
2. **三层影响分析切中要害**：轨迹展示、围栏告警、统计距离——这三者确实是 GPS 抖动在系统中传导的主要路径。
3. **Kalman 滤波的判断正确**：30 分钟采样间隔对于 Kalman 的预测-修正循环来说确实太稀疏，现阶段不上是对的。

## 几点建议和疑问

### 1. 15m 前端过滤阈值可能偏大

抖动直径约 11m（南北）× 9m（东西），平均中心距 RTK 真值仅 2.2m。如果前端用 15m 作为最小位移阈值，一头牛真的缓慢移动了 10-14m（比如在围栏边吃草移动），也会被合并为"静止"。建议：

- 阈值设为 **抖动直径的 1.2-1.5 倍**，即 **12-15m**，但 15m 是上界
- 或者更精确：不只看两点距离，而是看**连续 N 个点是否都在同一个聚类半径内**——这和第二层方案思路一致

### 2. "停留点"概念值得提前部分落地

第二层方案（停留点聚类）对围栏告警的改善可能比前端展示更大。dev 环境已经观察到 livestock 65 的 FENCE_APPROACH → FENCE_BREACH 自动升级循环——这个问题如果不修，用户会收到大量误报告警，比轨迹锯齿更致命。

建议考虑：**把围栏检测的抖动过滤提前到第一层**，可以和前端展示过滤一起做：

```
围栏检测时不直接用原始 GPS 点，而是先判断：
- 如果当前点距上一个"有效位置"（上次触发告警检查的位置）< 抖动阈值 → 跳过本次围栏检测
- 只有位移超过阈值时才重新计算围栏距离
```

改动量和前端过滤差不多，但解决的问题更关键。

### 3. 可以考虑后端加一层简单的滑动窗口中值滤波

这是介于"纯前端过滤"和"完整停留点聚类"之间的低成本方案：

- 在 GPS 数据查询/写入时，对最近 3-5 个点取中值（纬度、经度分别取 median）
- 不改变存储，只在读取侧做
- 对 30 分钟间隔的数据，3 点窗口 = 1 小时，平滑效果足够
- 比 Kalman 简单得多，比纯阈值过滤更稳定

### 4. 统计距离的修正方式

文档建议用停留点之间的距离替代原始点累加。这个方向对，但需要注意：如果一头牛真的在移动（比如从牧场北边走到南边 500m），中间没有停留，按停留点逻辑会低估实际移动距离。

建议统计距离分两段计算：
- **移动段**：连续位移超阈值的点之间，按 haversine 累加
- **停留段**：聚类为一个点，距离贡献为 0

这样既不过度统计抖动，也不低估真实移动。

### 5. 小问题

- 原文档第 33 行的坐标分布表中，出现次数为 8 的点 `(28.246587, 112.851677)` 出现次数明显少于其他点，可能是卫星几何恶劣时的离群值，值得标注。

## 优先级建议

**围栏告警过滤 ≥ 轨迹展示过滤 > 统计修正**

误报告警对用户信任的伤害最大，应该最优先处理。
